整体管道

从 profile + zone 数据 出发，经过 三个独立层 收敛到 5 张卡片：

profile + zone 数据  →  L1 召回  →  L2 打分  →  L3 LLM 精排  →  终选 5
                        (规则过滤)  (16 维加权)  (top60→5, D-046)

三个层之间的核心收敛比（lunch / shenzhen-bay / 默认 profile 快照, 2026-05-13）：

层输入规模输出规模主导决策
L1 召回11123 dish / 239 家2206 combos规则（profile + meal_log + meal_type）
L2 打分2206 combostop 6016 维加权和（确定性）+ 4 层 cap
L3 精排top 60top 5 (3+2)LLM 语义判断 + 规则 fallback
数字仅供示意：这些数量会随 profile.recall.per_restaurant_max、price_range.hard_max_*、preferences.* 黑名单等参数变化，实际跑一次看调试台 L1 stat-grid 才准。
当前快照：per_restaurant_max=20、hard_max_oil_level=4、hard_max_eta_min=45、hard_max_lunch=80。
调试心法：先看 L1 召回数量正不正常，再看 L2 top10 的 breakdown 找异常分量，最后看 L3 的 LLM payload/response。每一层都能独立判断对错。

L1 召回 · 5 步串联（D-041 后）

输入：zone 全量 dishes_tagged + restaurants + meal_log + meal_type。

D-041 在原来 3 步基础上加了 P0 餐厅级硬过滤（ETA / 名字）+ P1 菜级黑名单（主蛋白/烹饪方式/菜系）+ combo 级价格硬过滤。现在是 5 步串联。

① 餐厅级 ban · compute_extra_banned_restaurants D-041

遍历 restaurants.json，命中任一规则的整家餐厅都从候选池移除。

规则触发profile 字段
ETA 超硬上限delivery_eta_min > hard_max_eta_min（外卖只卡 ETA 不卡距离）delivery_constraints.hard_max_eta_min
名字命中黑名单餐厅 name + brand 拼起来子串匹配 avoid_restaurants（如 "烧烤"、"麦当劳"）preferences.avoid_restaurants
近期已吃近 no_same_restaurant_within_days（默认 7）天的 meal_log 出现过diversity.no_same_restaurant_within_days
三个集合并集 = avoid_rests，传给下一步 ②。调试台 L1 stat-grid 分别展示 ban·ETA / ban·avoid 名字 / ban·近期吃过 三个数字。

② 菜级 hard_filter（含 P1 三个黑名单）

一条条 dish 砍。命中任一就丢，并记录原因。D-041 加 3 行

规则触发profile 字段
所属餐厅被 ban来自 ① 的 avoid_rests—
命中 avoid_dishes菜名子串匹配（"红烧肉" 命中 "红烧肉饭" 也丢）preferences.avoid_dishes
辣度超上限spicy_level > spicy_tolerancepreferences.spicy_tolerance
油超硬上限oil_level > hard_max_oil_levelplate_rule.hard_max_oil_level
主蛋白黑名单main_ingredient_type ∈ avoid_main_ingredients 精确匹配preferences.avoid_main_ingredients
烹饪方式黑名单cooking_method ∈ avoid_cooking_methods 精确匹配preferences.avoid_cooking_methods
菜系黑名单cuisine ∈ banned_cuisines 精确匹配（如 ["饮品甜品"]）preferences.banned_cuisines
月销不达标monthly_sales > 0 AND < min_monthly_sales
（==0 视为无数据，不卡）		recall.min_monthly_sales
下架metadata.is_available == False—
banned_cuisines vs disliked_cuisines：前者硬过滤（召回阶段直接 ban，菜永远不出现），后者软扣分（L2 cuisine_preference 维度 -1）。

③ diversity_filter · 主蛋白多样性（菜级）

近 no_same_main_ingredient_within_days（默认 3）天吃过的主蛋白（仅 红肉 / 白肉 / 海鲜 / 豆制品）→ 砍该 main_ingredient_type 的菜。

纯素 / 主食 / 蛋 不受此约束。

④ build_combos · 两条路线

按餐厅分桶，每家最多生成 per_restaurant_max 个 combo。

路线 A · 完整套餐：盖饭/便当类（is_complete_meal=True）单菜 → 也可 +1 蔬菜补蔬。

路线 B · 笛卡尔积：蛋白菜 × 蔬菜 × （主食可选），覆盖三件套。

每个 combo 必须过 弱约束三件套（D-023）：

  • 至少 min_vegetable_dishes（默认 1）道蔬菜（vegetable_ratio≥0.6 或 main_ingredient_type=纯素）
  • combo 总蛋白 ≥ min_protein_g（默认 25g）

combo 去重：按 frozenset(dish_id) 去重，路线 A/B 撞车时只保留一份。

⑤ combo_price_filter · 总价硬过滤 D-041

按 meal_type 选阈值，combo 总价（所有菜 price 之和）超阈值 → 整 combo 丢弃：

  • meal_type=lunch → 用 price_range.hard_max_lunch（默认 80）
  • meal_type=dinner → 用 price_range.hard_max_dinner（默认 100）
价格双层：hard_max_* = 召回硬卡死；lunch_max/dinner_max = L2 软扣分（price 维度，超出线性扣分）。
调试台看哪：
L1 stat-grid 11 个数字：全量菜 → 三类 ban 数 → hard_filter 后菜 → diversity 后菜 → 出 combo 餐厅 → combos（价过滤前）→ drop·价超限 → 最终 combos。
「餐厅级 ban 明细」：D-041 ① 命中的餐厅（ETA / 名字）逐家列出。
「combo 价格硬过滤」：D-041 ⑤ 被丢的 combo 前 20 条，看总价超了多少。
「硬过滤丢弃原因聚合」：把所有丢弃 reason 里的数字归一成 N，统计条数排序，一眼看出"是辣度卡多 还是 主蛋白卡多"。

L2 打分 · 16 维加权和

每个 combo 算 16 个分量 × 各自权重 → 求和 = score。没有归一化，所以 score 是浮点累加（V2 默认权重下大约 1~6 区间）。

权重在 profile.scoring_weights 里配；缺字段 → 走 V2_DEFAULT_WEIGHTS（见 chisha/score.py）。

G3 · 履约 3 维（懒得跑远 / 不想等 / 别太贵）

维度怎么算默认权重

调试台看哪：
L2 section → 顶部权重 badge 行（profile_overrides 生效后这里会变）
top 30 表格 点 row 展开 → 右半边是 breakdown，按绝对值大到小排序，绿正/红负/灰零；一眼看出"这条凭啥排第 3"。
左半边是 combo 内每道菜的全字段（main_ingredient_type / cooking_method / oil / wetness / sweet / grain / 蛋白 g），方便对营养标签准不准。

D-042 · per-restaurant cap（防同店霸榜 top30）

打分排完序后，rank_combos → cap_per_restaurant(k=3)：每家餐厅最多保留前 3 个 combo，其余按全局分数顺位下放到尾部。

  • 动机：per_restaurant_max=20 让单店最多 20 个 combo 进入打分；分数相近的同店变体会一起挤在 top30，挤掉真正多样的候选；LLM 精排只看 top30，前面没多样性，后面也救不回来。
  • 实现：cap_per_restaurant 按 ranked 顺序遍历，每 restaurant_id 计数 ≥ k 后扔到 tail；返回 head+tail，不丢任何 combo。
  • K 来源：profile.recall.per_restaurant_top_k（默认 3）。生产 V1/V2/refine 三条路径都走 resolve_cap_k(profile)，缺字段或非法整数回退默认。k=0 passthrough（不 cap）。
  • 调试台显示：L2 stat-grid 显示 cap 前后的「top30 餐厅数」「单店最多 combo」对比，cap 起作用时数字会变颜色。
为啥不在召回阶段直接 cap？ 召回的 per_restaurant_max=20 是组合空间预算（避免一家店产出 200 个组合炸内存），cap 是排序后多样性（同店分数相近的 combo 别一起霸榜）。两者职责不同：前者裁原料、后者裁成品。

L3 LLM 精排 · top60 → 5 D-046 D-047 D-048

把 L2 的 top 60 + profile + context 拼成紧凑文本（每菜一行），让 LLM 选 5 个，强制 3 exploit + 2 explore。prompt 拆 system / user，system 走 Anthropic prompt cache 100% 命中。

D-047 重构（tool_use forced schema）：D-046.1 的 json_mode + regex 提取路径在 sonnet-4.6 / want_light 等复杂 mood 下会撞英文 CoT 占满 max_tokens 导致 fallback。D-047 改用 tool_use forced schema（Anthropic tool_choice={"type":"tool","name":...}，OpenRouter 走兼容协议），同时把 LLM 调用抽到共享 helper chisha/rerank.py:_run_llm_rerank，debug 与生产复用一份代码（消灭 D-046.1 双份漂移的根因）。默认 model 切到 claude-opus-4-7（V3+V4 实测 opus 比 sonnet 更尊重 taste）。

Provider 路由 · D-047

L3 走 chisha.llm_providers 抽象层，按下面的优先级解析 provider：

优先级来源说明
1CHISHA_LLM_PROVIDER env跑测试 / 临时切换用，覆盖一切
2profile.llm.provider持久化配置（推荐）
3auto-detect看 ANTHROPIC_API_KEY / OPENROUTER_API_KEY 谁存在

支持三个 provider：anthropic_api（直连，能用 cache_control: ephemeral）/ openrouter（OR 中转，须显式标 cache_control 才生效）/ claude_code_cli（本地 CLI 子进程，无 API key 时的兜底；不走 tool_use，system prompt 自动 patch 成"直接输出 JSON"）。

LLM 必须返回（tool_use schema · D-047）

定义在 chisha/rerank.py:_RERANK_TOOL_SCHEMA，以 Anthropic tools + tool_choice 强制结构化输出。模型必须调 submit_rerank 工具回填 candidates 数组；每条 candidate 必填以下字段，否则 _validate_llm_candidates 返回 None，整批 fallback。

字段类型校验
rankint1..n 连续
combo_indexint回指 top60 的 index，不重复，≥0
is_explorebool—
fit_scorefloat0.0 ~ 1.0 闭区间
taste_matchfloat / null0.0 ~ 1.0 或 null

health_flags 不再让 LLM 算（D-046）：veg_ok / protein_ok / oil_ok / processed_meat / sweet_sauce / wetness 这 6 个 bool 全是确定性规则可算的，由 rerank.py:_compute_health_flags 在拿到 LLM 输出后补齐。对外字段集和 V2 完全兼容。

失败路径 · fallback_rerank

LLM 输出无 JSON / 校验失败 / 调用异常 → 走规则版：

  • exploit：打分 top (n - n_explore) 走 diversify_top（品牌+菜系强制多样性）
  • explore：剩余里挑中段、近 7 天没吃过的 cuisine/method（_pick_explore）
  • 所有字段用最简规则填占位（one_line_reason 用 "汤水清爽，低油1.7，蛋白达标" 风格）
_enforce_brand_unique 兜底：LLM 选完后做一次品牌去重；若去重后不足 5 个，从 top60 里拿没用过品牌的补；这些补位条 fit_score 是原始 score（>1），one_line_reason 写"为多样性补位"。在调试台上看到 fit>1 / 理由是 "为多样性补位" 就是它。
调试台看哪（D-047 / D-048 更新后）：
L3 section header 一行同时带出 status / provider / model / latency。
stat 卡片新增：
  • provider = anthropic_api / openrouter / claude_code_cli，看实际生效路由
  • latency：opus 正常 ~12s；超 30s 多半是 OR 路由抖动或撞 CoT
  • system 字符 / user 字符（D-047 拆开了；老版只有 prompt_chars）
  • cached tok：anthropic_api / OR 都填 cache_read_input_tokens；首次调用 0，第二次起应该 ≈ system 段 token 数（约 1700）。如果一直 0，说明 cache 没起作用
  • stop_reason：必须是 tool_use（Anthropic）或 tool_calls（OR），stop 一般代表 LLM 没调 tool 直接输出文本，会走 fallback
折叠区：
  • 「LLM user message」~7K 字符（D-046 紧凑化, 旧版 ~38K）
  • 「LLM tool_input」= forced schema 解析后的 dict（D-047 新增）
  • 「LLM 原始返回」= provider 原始 raw_text
  • 「parsed_candidates」= 业务校验后的 5 条
status badge：成功 (ok) / 失败 / fallback / config_error / 未调用。
看到 config_error 橙框（D-048）：去修 CHISHA_LLM_PROVIDER / profile.llm.provider / ANTHROPIC_API_KEY，不要当 fallback 用。

Context 注入 · 软调权

ContextSnapshot（D-034）每次推荐前生成，不做硬过滤，只做软调权：

  • L2 阶段：context_boost 维度仅在 daily_mood = neutral（V1.2 唯一生效值）时进入打分；当日偏好统一走 refine 文本通道。
  • L3 阶段：整个 snapshot 拍进 LLM payload，由 LLM 用语义判断（如用户在 refine 里写"想喝汤"，LLM 倾向带汤水的）。

daily_mood 的 L2 效应（V1.2 简化后）

看 chisha.score.context_boost：

mood触发条件调分
neutral / None—（唯一生效值，默认无加成）0
want_soup / want_light / low_carb / want_clean / want_indulgentD-071 砍 4 条，D-073 进一步砍 want_soup — 全部不再产生 L2 加分（schema 仍接受入参以保证向后兼容）。当日偏好改走 RefineIntent.flavor_tags。0

历史背景：D-034 曾按 6 个 mood 给 ±0.2~0.5 的加权，D-071/D-073 砍掉以让"当日情绪偏好"统一走 refine 文本通道（防 mood chip + refine text 双通道冲突）。

main_ingredient_type红肉/白肉/海鲜/蛋/豆制品/纯素/主食/混合主蛋白来源；多样性过滤靠它
cooking_method清蒸/炖煮/凉拌/煎炒/油炸/烧烤/...多样性 / explore 用
oil_levelint 1~51 清淡 ~ 5 重油
spicy_levelint 0~30 不辣 ~ 3 重辣
protein_grams_estimatefloat预估蛋白克数
vegetable_ratio_estimate0~1≥0.6 算蔬菜菜
dish_role主菜/配菜/主食/汤/套餐组合校验 + carb_quality 用
processed_meat_flagbool火腿/培根/午餐肉/腊肠/香肠等
sweet_sauce_levelint 0~3糖醋/锅包肉/照烧风味，≥2 扣分
wetnessint 1~31 干 / 2 卤水 / 3 可喝汤底
grain_type糙米杂粮/全麦面/粗粮/粥/白米/精制面/...仅 dish_role=主食 时看；D-042 起"粥"不算 GRAIN_GOOD（精制白米煮，汤水价值已被 wetness 维度覆盖）
is_complete_mealbool盖饭/便当类，单菜就能构成主菜+主食

health_flags（LLM 返回 / fallback 都填）

字段含义"好"是哪个值

processed_meat含加工肉false
sweet_sauce含甜口false
