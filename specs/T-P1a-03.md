# T-P1a-03 · refine 多 slot LLM 解析层 + 安全带

参考: `docs/design_briefs/2026-05-18-refine-v2-faithful-refine.md` §4 + §5。

## What
1. 新建 `chisha/refine_intent_extractor.py`（或扩 `refine_intent.py`），一次 LLM 调用输出 §4 多 slot 结构：
   ```
   redirect: {cuisine_want/avoid, cuisine_candidates_expanded, ingredient_want/avoid,
              ingredient_synonyms, brand_avoid, cooking_method_avoid, food_form_avoid}
   constrain: {oil, price_max, quality_floor, delivery_only, max_distance_km, functional}
   reference: {reference_meal_id, relation}
   reject_previous: bool
   raw_understanding: str
   schema_version: "1.0" (T-00 已加)
   ```
2. 3 条安全带：
   - Schema 验证 + 失败降级到"空 refine 模式"，UI 友好提示「没听懂你说的，按你的日常偏好推了一组」。绝不崩。
   - Trace 双存：raw 原文 + 结构化结果 + `raw_understanding` 三份都写。
   - 冷启动 eval set：30-50 条人工标注 refine 文本（覆盖菜系切换/食材偏好/口味追加/品牌排除/相对参考/边界 case），写 runner 脚本。
3. 字段空洞务实降级（brief §5）：`quality_floor`/`delivery_only`/`max_distance_km`/`reference` 块 LLM 仍输出但 L1/L2 不消费，只透传给 L3 prompt，trace 标 `unsupported_in_recall=true`。
4. 兼容旧 `RefineIntent` 字段（向后兼容期）：保留旧字段名同时暴露新多 slot 结构。

## Why
- Faithful Refine 第一原则的执行核心：系统理解多深 = 用户信任能给多少。
- 修 brief §0 问题 1: "refine 自然语言诉求没被很好理解和满足"。
- 字段空洞策略：听懂但暂时做不到 > 假装没听见。

## Done When
- eval set 30-50 条全部跑通，准确率 ≥ 85%（人工标 → LLM 解析 → 比对 slot 命中）。
- 5+ 失败 case（LLM 返回坏 JSON / 超时 / schema 不匹配）100% 走降级路径不崩，UI 收到友好提示。
- trace 里能看到 LLM 原始 raw_understanding 文本。
- 已有 refine 用例全部通过（不破坏 D-073 / D-076 / D-079 的回归集）。
- profile.yaml 用 sonnet（避免 deepseek-flash 稀释 intent，见 memory `chisha_l3_model_refine_intent_sensitivity`）。
- `uv run pytest tests/ -q` 全绿。
- 空输入路径 baseline_l2_snapshot 0 diff。

## Affected files (预估)
- `chisha/refine_intent_extractor.py`（新建）或 `chisha/refine_intent.py` 大改
- `chisha/llm_client.py` / `chisha/llm_providers/`（如有 prompt 复用）
- `chisha/trace_store.py`（双存）
- 新建: `tests/refine_eval/eval_set.jsonl`（30-50 条）+ `scripts/refine_eval_runner.py`
- 测试: `tests/test_refine_intent_extractor.py`

## 红线
- 失败降级必须 100% 覆盖。LLM 出错时绝不崩，绝不瞎猜结果。
- LLM prompt 必须显式标 cache_control（memory `chisha_llm_call_principles`）。
- 不消费字段空洞字段（`quality_floor` 等）在 L1/L2 — 只能透传到 L3。

## 不做
- 不动 recall.py 召回逻辑（T-P1a-02 已经在做）。
- 不实现 reference 块的真实 resolver（那是 T-P2-01）— 本任务只解析出 slot，L1/L2 不消费。
- 不引入 tool_use 模式（L1 抽取低频，走 text+JSON 路径，见 memory）。
