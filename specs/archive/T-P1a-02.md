# T-P1a-02 · L1 召回参数 refine 分支重写

参考: `docs/design_briefs/2026-05-18-refine-v2-faithful-refine.md` §6 链路参数对照表。

## What
在 `chisha/recall.py` 加 `if intent is not None and intent.has_redirect()` 分支，refine 模式下：
1. `per_restaurant_max` 从 3 提到 5-8（具体值落地时小范围 grid）。
2. 总召回上限放宽 2-3x。
3. `ingredient_want` 进 L1 反查（含肉/含鱼/含菜的菜全召回，不只 L2 加 0.2 权重）。
4. `L3_INPUT_TOP_K` 加保底逻辑：「至少 30 个 intent 命中」不足时三级回落：
   - 精确（intent 全匹配） → 同源扩展（cuisine_candidates_expanded） → 全集
   - 每级 trace 落事件可查（用 T-00 的 trace event 框架）

## Why
- 解决 brief §0 问题 2: "trace 里送 L3 的 combo 数有时凑不满 60" → LLM 在不够 30 个候选上做精排。
- 修 brief 提到的"L1 召回数量参数不区分有无 intent"代码事实。
- ingredient_want 不再只是 L2 0.2 权重 → 真正进入召回方针。

## Done When
- 5 个典型 refine 文本（"想吃湘菜肉多一点"、"低油的"、"避开萨莉亚"、"想喝粥"、"吃辣"）实测送 L3 候选 ≥ 30 个相关 combo。
- 空输入模式行为完全不变：`uv run python -m scripts.baseline_l2_snapshot --out-dir tmp/baseline_traces_after && uv run python -m scripts.compare_traces` 0 diff。
- 三级回落有 trace 日志可查（每次回落都写 `recall_fallback_event`）。
- `uv run pytest tests/ -q` 全绿。
- 新增至少 5 个 e2e 用例覆盖 refine 召回参数 + 三级回落。

## Affected files (预估)
- `chisha/recall.py`（核心）
- `chisha/refine_intent.py` 可能需要加 `has_redirect()` 等便捷方法
- `chisha/trace_store.py`（写 fallback 事件，用 T-00 接口）
- 测试: `tests/test_recall_refine_branch.py`（新增）

## 红线
- 空输入路径（无 refine）必须 baseline_l2_snapshot 0 diff。
- ingredient_want 反查不能引入 N+1 查询导致召回耗时翻倍 — 加性能 sanity check。

## 不做
- 不动 L2 权重（那是后续单独事，brief §6 提到的 satisfaction 0.7/diversity 0.3 在 P2 簇式输出再调）。
- 不动 LLM 抽取层（T-P1a-03 才管）。
