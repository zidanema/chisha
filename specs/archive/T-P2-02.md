# T-P2-02 · 簇式输出（intent 内部子类多样化）

参考: `docs/design_briefs/2026-05-18-refine-v2-faithful-refine.md` §6（多样性纵向）+ §10 P2-02。

## What
refine 模式下，出 5 道菜按 intent 内部子类多样化（**纵向多样性**，不是横向 cap）：
- 湘菜 → 家常/小炒/腊味/剁椒 4 个子类
- 日料 → 拉面/寿司/居酒屋/定食/烤物 5 个子类
- 川菜、粤菜等同理
- 子类清单可配置（`config/cuisine_subtypes.yaml` 或类似）
- 实施: L2 加 `intent_subtype_diversity` 维度 或 L3 prompt 加纵向多样性约束

## Why
- 解决 D-073.1 副作用："cuisine_want 免 3 层 cap 后同品牌连锁日料刷屏"。
- 横向 cap（cuisine/brand/food_form）在 refine 模式下伤召回；改用纵向多样化是根治。
- Faithful Refine + 多样性兼顾 — 不在意图层稀释（仍推用户要的菜系），而是子类丰富。

## Done When
- 配置/数据：常见菜系（湘/川/粤/日/韩/泰 等）的子类清单完整（至少 8 个主菜系覆盖）。
- 实测 5 个典型 refine 表达，5 道结果子类 ≥ 3 个（湘菜 refine 出来不全是腊味，川菜 refine 出来不全是火锅）。
- D-073.1 测例：日料 refine 不出现同品牌连锁刷屏（萨莉亚式）。
- 空输入路径 baseline_l2_snapshot 0 diff。
- `uv run pytest tests/ -q` 全绿。

## Affected files (预估)
- `config/cuisine_subtypes.yaml`（新建或扩 methodology 配置）
- `chisha/score.py` 或新建 `chisha/diversity.py`
- `chisha/recall.py` 或 `chisha/rerank.py` 接入
- 测试: `tests/test_subtype_diversity.py`

## 红线
- 空输入路径必须 baseline_l2_snapshot 0 diff（纵向多样化只在 refine 模式生效）。
- 子类判定不引入新的 LLM 调用（走规则/词表/cuisine_method 推断）。
- 不破坏 D-073.1（cuisine_want 免 3 层 cap）— 横向 cap 在 refine 模式下仍豁免，本任务是叠加的纵向。

## 不做
- 不做横向 cap 调整（brief 已经在 D-073.1 处理过）。
- 不动 L1 召回逻辑（T-P1a-02 已经在做了 ingredient 反查）。
