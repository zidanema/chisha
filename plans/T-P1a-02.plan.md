### Step 2 · `ingredient_want` 进 L1 反查

refine 模式下,在 `recall()` 主流程 hard_filter 之后,扩充候选池:
```python
if has_intent:
```

### Step 3 · `_apply_intent_buckets` 三级回落 + trace

**Codex blocker #2 修正**: 30 是 floor, 不是 ceiling。L3 期待 60 输入,要努力填到 `L3_INPUT_TOP_K=60`。

当前 `_apply_intent_buckets` 已经分了 `bucket_exact / bucket_soft / bucket_rest`。改造成:
- 计算"intent 命中 combo 数" = `len(bucket_exact) + len(bucket_soft)`
- `target = L3_INPUT_TOP_K`  # 默认 60
- `floor = 30`               # brief §6 "至少 30 个 intent 命中"
- 三级回落判定 (每级仍尽可能填到 target=60):
  - **Level 1 (精确)**: `intent_hit ≥ floor` → 返 exact + soft + rest 拼到 target (intent 命中放前,rest 兜底; rest 数量上限不再卡 30)
  - **Level 2 (同源扩展)**: `floor > intent_hit ≥ 10` → 同 Level 1 (返 exact + soft + rest 至 target), 但 trace 标记 "intent_hit_below_floor" 警示
  - **Level 3 (全集兜底)**: `intent_hit < 10` → 返全集 (exact + soft + rest), 表明 refine 严重偏离数据
- **关键差别**: Level 1 vs 2 vs 3 不影响实际 combo 列表大小 (都尽量填到 60), 仅影响事件 label + 后续可观测性. brief §6 的"三级回落" 含义是 trace **告知** + L3 prompt 上层可见, 而非真砍掉候选
- 每级 append 一条 `recall_fallback_event` 到 trace:
  ```python
  {
    "event_type": "recall_fallback",
    "level": 1 | 2 | 3,
    "intent_hit_count": int,
    "total_returned": int,
    "ts": float,
  }
  ```

**问题**: `recall()` 不直接接 trace dict。事件落哪?
- 选项 A: 让 `recall()` 也接 kw-only `recall_fallback_events: list | None = None`,refine.py 调用方收集后 append 到 trace
- 选项 B: `_apply_intent_buckets` 内部 logger.info,trace 不持久化
- **选 A**: 与 T-00 hard_filter_events 设计一致,trace 完整性优先
