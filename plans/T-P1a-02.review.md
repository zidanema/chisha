# T-P1a-02 Adversarial Review

## 1. Reverse lookup performance

Severity: OK

`_ingredient_want_reverse_lookup()` is a linear scan over `dishes_tagged` (`chisha/recall.py:746-768`). The expensive safety checks are simple substring/set checks: L0-A loops configured allergies over dish text (`chisha/l0_constraints.py:66-79`), and L0-B checks one dietary law plus fixed halal keywords (`chisha/l0_constraints.py:82-106`). Actual tagged data is 11,123 dishes for `shenzhen-bay` and 2,117 for `home` (validated with the same dataset shape summarized by `scripts/validate_data.py:156-159`); local benchmark using `.venv/bin/python` with 7 ingredient wants, 6 allergies, and halal averaged 10.2ms / 2.0ms respectively.

Recommendation: No blocker at current scale. If data grows to true tens of thousands per zone, add a cheap precomputed ingredient/name index before this path.

## 2. Reverse lookup redundancy / new-dish behavior

Severity: OK

Not a logic error. `recall()` first runs full `hard_filter(dishes_tagged, profile, avoid_rests)` (`chisha/recall.py:664-665`), then reverse lookup scans the original full set and appends deduped hits (`chisha/recall.py:669-678`). Because `hard_filter()` drops preference/min-sales/availability failures after L0 and restaurant checks (`chisha/recall.py:280-309`), reverse lookup can surface dishes excluded by `avoid_dishes`, spice/oil, avoid ingredient/method/cuisine, `monthly_sales`, or availability while still respecting restaurant bans and L0-A/B (`chisha/recall.py:746-753`).

Recommendation: Update wording/tests if needed: this bypasses `hard_filter` dish-level preference gates and min-sales, but it does not bypass later main-ingredient diversity because `diversity_filter()` runs after append (`chisha/recall.py:680-681`) and can still drop recovered dishes (`chisha/recall.py:370-379`).

## 3. Three-level fallback intent

Severity: OK

The implementation is plan-conformant: all levels return `bucket_exact + bucket_soft + bucket_rest`, with `level` only changing event metadata (`chisha/recall.py:833-866`). The approved plan explicitly says Level 1/2/3 do not affect actual combo list size and are only an event label / observability signal (`plans/T-P1a-02.plan.md:67-72`), and the v2 plan audit approved that interpretation (`plans/T-P1a-02.plan-review-v2.md:7-15`).

Recommendation: None for T-P1a-02.

## 4. `profile.yaml` default / override discoverability

Severity: WARN

Code supports `profile.recall.refine_per_restaurant_max` with default `5` (`chisha/recall.py:689-694`), and `profile.yaml` has a `recall:` section (`profile.yaml:134-147`). However, `refine_per_restaurant_max` is not present there, despite the approved plan listing `profile.yaml` as an affected file and saying to add optional `recall.refine_per_restaurant_max=5` (`plans/T-P1a-02.plan.md:133-138`). The only in-code discoverability is a source comment, not user-facing profile documentation (`chisha/recall.py:689`).

Recommendation: Add a commented/default `refine_per_restaurant_max: 5` entry under `profile.yaml:134-147` in a follow-up, or explicitly document that omission as intentional.

## 5. Fallback event timestamp

Severity: BLOCKER

Confirmed sandbox-time bug. `_apply_intent_buckets()` records fallback event timestamp with wall-clock `time.time()` (`chisha/recall.py:849-862`). Business trace timestamps elsewhere use `chisha.clock` when sandbox time matters, e.g. meal log timestamps use `clock.now_utc(root=root).isoformat()` (`chisha/recall.py:94-123`), and `clock.py` defines sandbox-aware `now()` / `now_utc()` (`chisha/clock.py:26-41`). Since these events are persisted into `trace.l1.recall_fallback_events` (`chisha/web_api.py:195-200`), sandbox replay/time-travel traces will carry real wall-clock seconds.

Recommendation: Use a sandbox-aware timestamp source for this trace event, ideally `clock.now_utc(...).isoformat()` if trace timestamps can be ISO strings, or pass an injectable timestamp/root through the recall event path.

## 6. Test strictness

Severity: WARN

`test_recall_with_intent_uses_refine_per_rest_max()` says the fixture should make one restaurant emit 5 combos, but asserts only `len(out) <= 5` (`tests/test_recall_refine_branch.py:111-124`). The fixture has 10 protein dishes plus 1 vegetable (`tests/test_recall_refine_branch.py:113-118`), and `build_combos_for_restaurant()` returns `uniq[:per_rest_max]` after generating protein+vegetable combinations (`chisha/recall.py:590-629`), so the current data does guarantee 5 combos.

Recommendation: Tighten the assertion to `len(out) == 5` so the test actually catches failure to use `refine_per_restaurant_max`.

Final VERDICT: BLOCKED, blocking issue: 5
