# T-P1a-03 Scaffold Plan Review

## 1. Scaffold scope reasonableness

The scope is reasonable for "Phase 1 ship now, full implementation later".

The plan correctly isolates V2 in a new module, keeps `chisha/refine_intent.py` and `chisha/refine.py` untouched, defers prompt redesign/eval/downstream consumption, and preserves the baseline 0-diff contract by not putting V2 on the production path.

This matches brief §4's intended direction without pretending to deliver the full multi-slot LLM parser or §6 downstream recall changes.

## 2. `from_legacy()` field preservation

Not yet sufficient as written.

V1 source of truth is `RefineIntent` in `chisha/refine_intent.py`, with these fields:

1. `cuisine_want`
2. `cuisine_avoid`
3. `ingredient_want`
4. `ingredient_avoid`
5. `cooking_method`
6. `flavor_tags`
7. `raw_flavor`
8. `portion`
9. `staple_preference`
10. `price_band`
11. `freeform_note`
12. `raw_text`
13. `schema_version`

The planned V2 shape has natural homes for `cuisine_*`, `ingredient_*`, and `raw_text`, but it does not specify a lossless home for `cooking_method`, `flavor_tags`, `raw_flavor`, `portion`, `staple_preference`, `price_band`, or `freeform_note`.

For scaffold correctness, the minimum fix is not to invent final semantics. Add an explicit preservation path, for example `legacy_v1: dict[str, Any] = field(default_factory=dict)`, populated from `intent.to_log_dict()` in `from_legacy()` and emitted by `to_log_dict()`. Then tests can assert no v1 key disappears while downstream remains unchanged.

`raw_understanding` can include `freeform_note`, but it should not be the only preservation mechanism because it is not structured and cannot losslessly preserve enum/list fields.

## 3. Minimum schema validation helper

The validation helper should stay shallow and structural. Do not block the scaffold for not enforcing final enum semantics.

Minimum checks that catch real bugs:

1. Top-level input is a dict.
2. `schema_version == "2.0"`.
3. `redirect` is a dict and all known redirect slots are lists of strings.
4. `constrain` is a dict.
5. `reference` is either `None` or a dict.
6. `reject_previous` is a bool.
7. `raw_understanding` and `raw_text` are strings.
8. `unsupported_in_recall` is a list of strings.
9. If `legacy_v1` is added, it is a dict.

Avoid over-engineering in this task: do not require final allowed values for `quality_floor`, `reference.relation`, functional flags, or inferred cuisine expansions. Those belong with the real V2 prompt and eval set.

## 4. `unsupported_in_recall` capture

The plan captures the mechanism directionally, but it needs tighter wording.

Brief §5's unsupported mechanism is about fields the system can understand but cannot faithfully execute in L1/L2 due to missing data or resolver support:

1. `constrain.quality_floor`
2. `constrain.delivery_only`
3. `constrain.max_distance_km`
4. `reference`

The plan's `unsupported_in_recall: list[str]` is a better scaffold shape than a single boolean because it can identify the unsupported field names. However, the plan should avoid marking every new unconsumed V2 slot as "unsupported" just because this scaffold does not wire V2 downstream yet.

For this scaffold, acceptable behavior is:

1. Keep all V2 slots present.
2. Keep L1/L2 consumption disabled.
3. Populate `unsupported_in_recall` with the known §5 field voids when those fields are present or, if slots are always empty in scaffold, document the known unsupported field names in `to_log_dict()`.
4. Do not label `brand_avoid`, `cooking_method_avoid`, or `food_form_avoid` as data-layer unsupported; §5 says they are usable once downstream is wired.

## 5. Breaking change risk for current V1 callers

Breaking risk is low if the plan is followed literally.

Current callers import `RefineIntent` and `parse_refine_intent` from `chisha/refine_intent.py`; the plan says not to edit that file, not to edit `chisha/refine.py`, and not to connect V2 to production. That preserves current V1 behavior and the baseline 0-diff guarantee.

The main guardrail is to keep V2 import direction one-way: `refine_intent_v2.py` may import V1 for migration, but V1 should not import V2. Tests should include at least one assertion that `parse_refine_intent(..., use_llm=False)` still returns a V1 `RefineIntent` with `schema_version == "1.0"`.

## Blocking issues

Plan section `Step 1 · chisha/refine_intent_v2.py` / field `from_legacy`: the declared V2 dataclass has no specified lossless storage for V1-only fields `cooking_method`, `flavor_tags`, `raw_flavor`, `portion`, `staple_preference`, `price_band`, and `freeform_note`. Add a structured legacy preservation field or an explicit field-by-field mapping that proves all V1 fields survive `from_legacy()` and `to_log_dict()`.
