### BLOCKING — 3 endpoints missing `_with_sandbox_sid` dep per plan §4.3

Plan §4.3 explicitly lists these as required to receive the dep:

1. **`PUT/POST /api/profile` (`api_put_profile`)** — write path. Calls `_profile_path()` → `data_root.profile_path(ROOT)` which uses ContextVar. Without the dep, `X-Session-Id` is silently ignored and the profile write goes to the wrong bucket (or prod). Verified: function signature at line 516 has no `_sid: str | None = Depends(_with_sandbox_sid)`.

2. **`POST /api/long_term_prefs/refresh` (`api_long_term_prefs_refresh`)** — write path. Calls `extract_and_save` / `bootstrap` with `root=ROOT`. Without the dep, sid-routed L1 extraction silently writes to wrong path. Verified: line 797-798, no dep.

3. **`GET /api/feedback/{session_id}` (`api_feedback_session`)** — read path. Calls `feedback_store.load_store(ROOT)` → `data_root.feedback_store_path(ROOT)` which uses ContextVar. Lower risk (read-only), but inconsistent with plan intent. Verified: line 751-752, no dep.

No test in the new test files covers these three endpoints with `X-Session-Id` headers.

The two write-path gaps (profile + long_term_prefs/refresh) represent silent-fallback risk: a client expecting sandbox isolation on these writes would silently get wrong-bucket writes. This directly contradicts the plan's §4.4 "fail-loud 保证" intent for the dep-applied endpoint set.
