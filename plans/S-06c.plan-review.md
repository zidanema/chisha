### Issue 1 — BLOCKER: per-session state.json never initialized by create_session

**File**: `chisha/sandbox.py:335` (`create_session`) + `chisha/sandbox.py:442` (`_state_path_for_sid`)

`sandbox.create_session(sid)` only does `bucket.mkdir()` — it does NOT write a `sessions/{sid}/state.json`. The `advance_meal` function reads `_state_path_for_sid(sid)` which for non-default sid resolves to `logs/sandbox/sessions/{sid}/state.json`, and raises `RuntimeError("sandbox state not found")` if the file is absent.

Plan §3.2 says the `/recs` endpoint reads `state.current_meal_idx`, and the eat/skip endpoints call `sandbox.advance_meal(sid=sid)`. Both will hard-fail (404/500) on any non-default session that was created via POST `/api/sandbox/sessions` — because that endpoint only creates the directory, not the state file.

The S-06b plan (line 182, test note) explicitly says "本任务不动 `sandbox.create_session`, 测试自行 mkdir + 写 state.json." This was a test-only workaround that was never resolved at the production API level.

**Impact**: Every non-default sandbox session created via the existing S-06a API will fail all S-06c endpoints. The plan's test setup (`手工写 s1/state.json`) masks this because tests bypass `create_session`.

**Fix required**: Either `create_session` must write an initial `state.json` with `enabled=True, current_meal_idx=0, total_meals=14`, or the S-06c session-init endpoint must do it, or the `/recs` endpoint must initialize it on-demand if missing. The plan must specify which.

### Issue 2 — BLOCKER: swap via `_impl_refine(refine_text="")` violates CONTRACTS.md invariant + creates orphan trace round

**File**: `chisha/web_api.py:170` (`_impl_refine`) + `docs/CONTRACTS.md:54`

The plan's swap path calls `_impl_refine(RefineReq(session_id=..., refine_text="", excludeIds=exclude_ids))`. Two sub-problems:

**2a**: `_impl_refine` unconditionally calls `refine_session(user_input="")`. CONTRACTS.md line 54 states: "refine 与空输入必须走差异化 L1 召回分支". The swap is not a user-expressed refine intent at all; calling the full `refine_session` chain with empty text for a swap is semantically incorrect — it increments `state.round`, appends an empty string to `state.refine_history`, and writes a `append_round` trace entry to trace_store (web_api.py:273). This pollutes the D-039 session trace with spurious "swap" rounds that look like user refines.

**2b**: `refine.py` does NOT accept `excludeIds` in its function signature (`def refine(session_id, user_input, profile, rests, tagged, meal_log, root, today, n, use_llm)`). The `RefineReq.excludeIds` field exists in the Pydantic model but `_impl_refine` never passes it to `refine_session`. So the exclude_ids on swap is silently dropped — swap will return results that still include the candidates the user wanted to exclude. The plan's §3.5 claim that "refine.py 原生支持 excludeIds" is factually wrong: `grep -n "excludeIds" chisha/refine.py` returns zero results.

**Fix required**: Swap should NOT use `_impl_refine`. It should call `recommend_meal` then post-filter exclude_ids from candidates, or a new lightweight helper. The plan acknowledges this as an option in §3.5 but chose the wrong path.

### Issue 3 — BLOCKER: `_build_decision_async` references `_impl_l1_refresh` which does not exist

**File**: `chisha/web_api.py` (entire file, grep result confirms)

Plan §3.3 BackgroundTask step 2a says: "真触发 L1 抽取: with `_sandbox_ctx(sid)`: ... `_run_l1_extract_internal(ROOT)` (复用 api_long_term_prefs_refresh 主体, 提抽出 `_impl_l1_refresh()`)". Plan §9 step 3 also says "复用 _impl_long_term_prefs_refresh 内部".

Neither `_impl_l1_refresh` nor `_run_l1_extract_internal` nor `_impl_long_term_prefs_refresh` exist in `chisha/web_api.py`. The current implementation at lines 921-963 uses a closure `_worker()` inside `_trigger_l1_extraction_async` with the L1 extraction logic inlined. This code is not extractable as a module-level function without refactoring.

The plan says "提抽出 `_impl_l1_refresh()`" implying S-06c would create this helper by refactoring the existing async worker. This is unplanned scope in a "not动" file boundary violation — plan §2 says `api_long_term_prefs_refresh` is not in the affected files list but the refactor would touch exactly that function's internals. The plan must be explicit that this refactor is part of S-06c scope and will be tested.

**Fix required**: Plan must explicitly list extracting `_impl_l1_refresh()` from `_trigger_l1_extraction_async._worker` as a step, and confirm the new helper preserves the existing lock semantics (`_L1_EXTRACTION_LOCK`) or deliberately drops them (with justification, since the lock exists to prevent concurrent LLM calls from corrupting `long_term_prefs.json`).

### Issue 4 — HIGH: ContextVar in BackgroundTask — plan's mitigation is stated correctly but incomplete for `load_profile` / `data_root` paths

**File**: `chisha/web_api.py:111` (`_sandbox_ctx`) + `chisha/sandbox_context.py:32`

The plan correctly identifies that ContextVar does not propagate across `threading.Thread` / thread pool, and says "in-function 用 `with _sandbox_ctx(sid):` 包". This is correct and sufficient for `recommend_meal`/`refine_session` calls.

However, inside `_build_decision_async`, the plan calls `_impl_refine` / `_impl_accept` which internally call `_profile_path()` (web_api.py:44) which calls `data_root.profile_path(ROOT)`. `data_root.profile_path` reads `current_sandbox_session()` from ContextVar. If the BackgroundTask thread doesn't have the ctx set before calling these helpers, `profile_path` returns the default (prod) profile, not the sandbox session's profile.

The plan's `with _sandbox_ctx(sid):` wrapping of the BG task body should handle this — BUT the mock=True path still calls `build_decision(...)` which eventually calls `sandbox.meal_idx_to_slot` (stateless, OK). The real risk is only in the non-mock path, which the plan correctly defers to S-08. So this is a documented risk, not an immediate blocker, but the plan should explicitly note that `_profile_path()` inside BG task must be inside the `with _sandbox_ctx(sid):` scope.

**Status**: Medium — properly mitigated for mock path; real path deferred to S-08 where it needs explicit verification.

### Issue 5 — HIGH: `build_decision` expects `picked_rec["name"]` but `format_v2_candidate` uses `restaurant.name`

**File**: `chisha/sandbox_decision_diff.py:77` + `chisha/api.py:66-75`

`build_decision` at line 77 does `name = picked_rec["name"]` (KeyError if absent). But `format_v2_candidate` (api.py:66-75) produces candidates where the name is at `candidate["restaurant"]["name"]`, not `candidate["name"]`. The plan says `_build_decision_async` receives `picked_rec` from `last_recs.candidates[rec_rank-1]` which stores v2 candidate dicts. So `picked_rec["name"]` will KeyError at runtime on non-mock paths.

For mock paths, `MOCK_CURRENT_RECS` are Rec view-model dicts (plan §5). The Rec shape has `name` at top level (per sandbox.ts types). But the v2 candidates stored in `last_recs.candidates` use the `format_v2_candidate` shape. The plan conflates the two shapes when passing to `build_decision`.

**Fix required**: Plan must specify that `build_decision` receives either (a) the Rec view-model dict (from `format_v2_to_rec`) or (b) the raw v2 candidate, and the `picked_rec["name"]` access must match. One or the other must be passed consistently, and `build_decision` or the adapter must be updated.

### Issue 6 — HIGH: `_impl_accept` called with D-039 `recommend_session_id` which has 24h TTL in sandbox sessions

**File**: `chisha/session.py:15` (TTL = 24h) + plan §3.3 step 6

The plan calls `_impl_accept(AcceptReq(session_id=recommend_session_id, ...))`. In a real sandbox session, the user may do `/recs` at D1-lunch and not eat until D1-dinner (they skipped lunch then came back). If the recommend_session_id was created >24h ago in wall-clock time, `load_session` returns None → `_impl_accept` raises HTTPException 404.

In sandbox time-travel, virtual clock advances but wall-clock doesn't. A user could simulate a full 7-day session in 30 minutes, with recommend_session_ids that are all well within the 24h wall-clock TTL. But if they pause for >24h in the middle, the D-039 sessions expire. The plan doesn't address this TTL mismatch between sandbox virtual time and real TTL.

**Fix required**: Document this as a known limitation, or use `check_expiry=False` in the sandbox eat path.

### Issue 7 — HIGH: `/recs` at done state (cur_idx == total_meals) — plan says 409 but only for eat endpoint; /recs is silent

**File**: Plan §3.2 (recs endpoint) vs §3.3 (eat endpoint done sentinel)

Plan §3.3 says "eat 端点先检查 cur_idx == total_meals → 409 'session done'". But plan §3.2 (`/recs`) has no equivalent done check. A frontend that calls `/recs` when `cur_idx == 14` will get `meal_idx_to_slot(14)` called — which returns `("dinner", 8)` (no error, since `idx < 0` is the only check). Then `recommend_meal(meal_type="dinner")` fires a full LLM call for a "ghost" meal that doesn't exist in the 7-day simulation.

**Fix required**: `/recs` must also check `cur_idx >= total_meals` and return 409 "session done" before calling recommend_meal.

### Issue 8 — MEDIUM: `history.json` append race between two /eat calls on same sid from different browser tabs

**File**: Plan §8 risk table (accepted as "S-07 上锁")

The plan acknowledges this but accepts it under "single-user assumption". The risk is real: two browser tabs making simultaneous `/eat` calls on the same sid could both read the same `history.json`, both append, and the second write overwrites the first's entry. Result: one meal permanently lost from history.

Given this is a "high risk" task and `history.json` is the source-of-truth for the D panel and future rollback (S-07), silent data loss here could corrupt the subsequent session state irreparably. The plan should at minimum use a file-level lock (e.g., `fcntl.flock`) for the read-modify-write cycle, or document that the `/eat` endpoint must be idempotent/serialized at the HTTP layer.

**Status**: Accepted risk per plan; noted here because it's not adequately mitigated for a "high risk" task designation.

### Issue 9 — MEDIUM: atomic write missing for `last_recs.json` and `decisions/{idx}.json`

**File**: Plan §8 risk table + §6.1/6.4

Plan §8 says "tmp + os.replace for meal_to_trace and history" but is silent on `last_recs.json` and `decisions/{idx}.json`. `last_recs.json` is read in eat/swap/refine — a partial write here (e.g., process killed mid-write) would cause eat to read corrupt JSON and crash. `decisions/{idx}.json` is the output of the expensive BG L1 extraction; losing it to a partial write is data loss.

**Fix required**: All four files should use atomic `tmp + os.replace`. The plan should state this explicitly for all 4, not just 2.

### Issue 10 — MEDIUM: `test_eat_concurrent_sids_isolated` uses asyncio.gather but the endpoints are sync FastAPI routes

**File**: Plan §7.2 test description

Plan says "asyncio.gather 两个 sid 不串" for the concurrency isolation test. FastAPI sync routes run in an anyio thread pool, not as async coroutines. `TestClient` is synchronous. You cannot use `asyncio.gather` with a synchronous `TestClient`. The test as described would not compile/run — you'd need `ThreadPoolExecutor` to truly test concurrent sync route isolation. The plan mentions "asyncio.gather or ThreadPoolExecutor" but asyncio.gather won't work with sync TestClient calls.

**Fix required**: Concurrency test must use `ThreadPoolExecutor` + `concurrent.futures.wait`, not `asyncio.gather`.

### NEW HIGH — Issue N2: `_ensure_not_done` 读 state.json，但 `_sandbox_ctx` 未在调用前 set

**File**: Plan §3.2/§3.5/§3.6，`chisha/sandbox_context.py`，`chisha/web_api.py:111`

### NEW HIGH — Issue N3: `_build_decision_async` 中 `with _sandbox_ctx(sid)` 包裹 `_impl_l1_refresh` — 但 `_L1_EXTRACTION_LOCK` 是模块级全局，BG task 会绕过 trylock

**File**: `chisha/web_api.py:929` (`_L1_EXTRACTION_LOCK`) + plan §3.3 BG task step 2a

修订 C 说 BG task 调 `_impl_l1_refresh(root, force_no_llm=False)` 走**非锁路径**（"BG task 走非锁路径"）。这与 CONTRACTS.md §Sandbox 第3条冲突：

> "sandbox 生命周期：reset/disable 先抢 `_L1_EXTRACTION_LOCK`（`_block_until_l1_idle_or_409`），否则 worker 中途 `save_prefs` 会污染 prod `long_term_prefs.json`。"

`_trigger_l1_extraction_async` 用 trylock + 设置 LOCK 保护整个 `_worker` 生命周期（`finally: _L1_EXTRACTION_LOCK.release()`）。如果 `_impl_l1_refresh` 是从 `_worker` 提抽出来的函数体，它就**不再 hold `_L1_EXTRACTION_LOCK`**。BG task 调 `_impl_l1_refresh` 时不持锁，这样 reset/disable 在 `_block_until_l1_idle_or_409` 里拿锁会立即成功（因为 BG task 不持锁），reset 可能在 `_impl_l1_refresh` 中途的 `extract_and_save` 写盘期间清空 sandbox 目录，导致 **write-after-delete** 并污染 prod data_root（如果 sandbox disabled 后 data_root 降级到 prod）。

CONTRACTS.md 精确规定了 lock 的目的就是防止这个 race。

**Fix required**: `_impl_l1_refresh` 提抽后，BG task 调用点**必须在 `_L1_EXTRACTION_LOCK.acquire()` 保护下**，或 `_impl_l1_refresh` 内部获取 lock。计划必须说明 lock 如何在提抽后保持语义。当前计划 §0 修订 C 说"trylock 不变，BG task 走非锁路径"，这违反了 CONTRACTS.md 要求。

### NEW MEDIUM — Issue N5: `_validate_and_route_sid` 调 `sandbox.is_enabled(ROOT)` 检全局 state，非 per-session，语义错误

**File**: Plan §3.1 `_validate_and_route_sid`

§3.1：
> "非 default: 校验 sid 格式 (_validate_sid); 校验 `sandbox.is_enabled(ROOT)` (409); 校验桶存在 (404)"

`sandbox.is_enabled(ROOT)` 读的是 `logs/sandbox/state.json`（全局 sandbox layout 开关）。一个非 default session 的 state.json（`sessions/{sid}/state.json`）中也有 `enabled` 字段（修订 A seed 时写 `"enabled": True`）。全局 sandbox 可以 enabled 而 per-session enabled 字段也应为 True，两者含义不同：

### 4. 修订 D (BG ContextVar): _sandbox_ctx across thread boundary
