# T-P1a-02 Plan Audit

## Check 1: per_restaurant_max side effects
WARN

Live code separates two caps:
- L1 combo generation uses `per_rest_max` only inside `build_combos_for_restaurant(..., per_rest_max, ...)`, and returns `uniq[:per_rest_max]` (`chisha/recall.py:525-529`, `chisha/recall.py:621-629`).
- `recall()` currently reads `profile.recall.per_restaurant_max` once (`chisha/recall.py:674`) and passes it to combo generation (`chisha/recall.py:681-683`).
- L2 `apply_caps()` is a separate ranked-combo cap. It resolves `per_restaurant_top_k` default 3 and `per_brand_top_k` default 2 (`chisha/score.py:1049-1065`), then drops anything over restaurant/brand/cuisine/food_form limits (`chisha/score.py:1117-1128`). D-049 is head-only: dropped tail is not returned (`chisha/score.py:1080-1082`).

So, if refine L1 raises `per_restaurant_max` from 3 to 5, `apply_caps()` will still correctly cap ranked output. However, brand cap can conflict with the intended benefit: default `per_brand_top_k=2` means at most 2 non-exempt same-brand combos survive L2 even if L1 emitted 5 per restaurant. Cuisine-want has an exemption from brand/cuisine/food_form caps (`chisha/score.py:1086-1090`, `chisha/score.py:1097-1127`), but ingredient-want has no equivalent exemption.

Plan coverage is incomplete. It says "D-073/D-073.1 cap 行为不动 (本任务不碰 score.py apply_caps)" (`plans/T-P1a-02.plan.md:133`) and "`不动 L2 weight / apply_caps`" (`plans/T-P1a-02.plan.md:153`), but it does not explicitly account for `per_brand_top_k=2` suppressing ingredient-want penetration after the new L1 expansion.

## Check 2: 三级回落 ≥30 vs ==30 (BLOCKING if wrong)
FAIL

The plan incorrectly treats 30 as a return target instead of a floor under `L3_INPUT_TOP_K=60`.

Live code confirms L3 takes top 60:
- `L3_INPUT_TOP_K = 60` (`chisha/rerank.py:47`).
- Refine slices ranked candidates with `top_k = ranked[:L3_INPUT_TOP_K]` (`chisha/refine.py:115-116`).

Exact plan text:
- "`**Level 1 (精确)**: intent 命中 ≥ 30 → 仅返 exact + soft (rest 不参与, 集中给 L3)`" (`plans/T-P1a-02.plan.md:65`).
- "`**Level 2 (同源扩展)**: intent 命中 < 30 但 ≥ 10 → 加 rest 的前 N 个保到 30 合计`" (`plans/T-P1a-02.plan.md:66`).
- Test name "`test_l1_recall_top30_level1_exact_only`: 注 ≥ 30 个 exact 命中,返回不含 rest" (`plans/T-P1a-02.plan.md:114`).
- Test name "`test_l1_recall_level2_fallback_to_30`: exact+soft 仅 15 个, level 2 触发回落补到 30" (`plans/T-P1a-02.plan.md:115`).
- Regression note: "`level 1 时 < 60 但本来命中就充足是好事;不动 L3_INPUT_TOP_K 常量`" (`plans/T-P1a-02.plan.md:141`).

This violates the brief requirement: ≥30 is the floor, not the final size. Level 1 should keep exact + soft first, then fill with partial rest up to 60 when available. The current plan would starve L3 of available diversity by returning exact + soft only, and Level 2 explicitly fills only to 30 instead of 60.

## Check 4: baseline 0-diff guard
WARN

The intended call path is mostly correct, but the plan's guard is not exactly the requested condition.

Live code:
- `recall()` signature defaults `intent=None` (`chisha/recall.py:632-640`).
- Current refine caller passes `intent=intent if not intent.is_empty() else None` (`chisha/refine.py:99-102`), so empty refine currently reaches `recall()` as `None`.
- Current `recall()` enters intent buckets on `if intent is not None:` (`chisha/recall.py:693-695`).

Plan text:
```python
has_intent = intent is not None and not (
    hasattr(intent, "is_empty") and intent.is_empty()
)
```
(`plans/T-P1a-02.plan.md:18-20`)

This does protect `None` and normal empty `RefineIntent`, and the refine caller already passes empty intent as `None`. But the requested guard was exactly "intent is None or intent.is_empty()". The plan uses `hasattr`, so an unexpected non-None object without `is_empty()` would enter the refine branch. That is probably acceptable for current callers, but it is not the exact baseline 0-diff guard requested.
