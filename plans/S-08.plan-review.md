## Iter 1

Adversarial audit against the current codebase, after reading `docs/CONTRACTS.md`.

VERDICT: BLOCKED

1. After `/eat` or `/skip`, the plan refreshes only `GET /api/sandbox/sessions/{sid}` and therefore leaves the next meal with zero recommendations. The backend deliberately deletes `last_recs.json` after eat/skip, and `FullSnapshot` only returns `currentRecs` from that file if it exists (`chisha/web_api.py:1530`, `chisha/web_api.py:1618`, `chisha/web_api.py:1894`, `chisha/web_api.py:1901`). The plan's online flow says eat polling done -> `getFullSnapshot` -> `setStore`, and skip -> `getFullSnapshot` (`plans/S-08.plan.md:74`, `plans/S-08.plan.md:75`), so acceptance item "history+1 + 新 recs" will fail (`specs/S-08.md:67`). Fix: after any meal advance (`eat`, `skip`, rollback/branch to an unfinished meal when snapshot has no recs, and initial load with empty `currentRecs`), call `POST /recs?mock_recommend=1` and merge returned `currentRecs` before rendering the active decision area.

2. The plan cannot surface backend `activeRules`, `taste`, `keywords`, `recent`, or `fatigue` with the stated affected-file set because `RightCol` hardcodes all right-panel data from `MOCK`, ignoring `useSandbox` output (`apps/sandbox-lab/src/components/RightCol.tsx:3`, `apps/sandbox-lab/src/components/RightCol.tsx:35`, `apps/sandbox-lab/src/components/RightCol.tsx:36`, `apps/sandbox-lab/src/components/RightCol.tsx:37`). The plan claims adapter/store will carry `activeRules` from backend refine (`plans/S-08.plan.md:58`, `plans/S-08.plan.md:63`, `plans/S-08.plan.md:77`) but only allows changing `App.tsx` and `useSandbox.ts` plus new api files (`plans/S-08.plan.md:6`, `plans/S-08.plan.md:11`). This breaks the cross-file FullSnapshot contract and the user-visible refine state: backend returns refine rules (`chisha/web_api.py:1774`, `chisha/web_api.py:1777`) but B panel will keep showing stale mock rules. Fix: include `RightCol.tsx` in scope and pass `activeRules`/profile panels as props, or explicitly defer those fields and remove the adapter/store claims.

3. The `SessionMeta.origin` mapping is missing. Backend `SessionMetaFull.origin` is `"blank"` by default (`chisha/web_api.py:1820`, `chisha/web_api.py:1884`), while the sandbox-lab frontend type only accepts Chinese literals `"真实快照" | "空白"` (`apps/sandbox-lab/src/types/sandbox.ts:15`, `apps/sandbox-lab/src/types/sandbox.ts:21`). The plan says backend meta is "几乎对齐" and only calls out `id` vs `sid` (`plans/S-08.plan.md:18`, `plans/S-08.plan.md:19`); `backendFullSnapshotToStore` likewise omits origin normalization (`plans/S-08.plan.md:58`). This will either fail typecheck or leak an invalid origin value into `TopBar`, depending on implementation. Fix: adapter must map `"blank" -> "空白"` and `"real_snapshot"`/other real origins -> `"真实快照"` with a tested fallback.

4. The plan's "mock_recommend=1 always" decision bypasses the real refine chain in every backend-online sandbox-lab run. Backend `/refine` only calls `_impl_refine` in the non-mock branch (`chisha/web_api.py:1727`, `chisha/web_api.py:1734`, `chisha/web_api.py:1737`, `chisha/web_api.py:1742`); with mock mode it just returns fixed mock recs and copies the user's text into `intent` (`chisha/web_api.py:1729`, `chisha/web_api.py:1730`). `CONTRACTS.md` requires Faithful Refine to be the top priority and failures to be explicit, not silently swapped for similar-looking behavior (`docs/CONTRACTS.md:9`, `docs/CONTRACTS.md:12`, `docs/CONTRACTS.md:13`). The plan changes spec-triggered mock mode into unconditional behavior (`plans/S-08.plan.md:81`, `plans/S-08.plan.md:106`), so S-08 would not validate whether the actual refine pipeline remains faithful. Fix: keep `mock_recommend=1` as an explicit dev/test switch for acceptance, but preserve a real-refine path (query/env/localStorage) and add at least one manual or automated real-refine smoke that verifies `_impl_refine` is reachable.

5. Test coverage is insufficient for the highest-risk new boundary. The plan adds three new TS API/adapter files (`plans/S-08.plan.md:7`, `plans/S-08.plan.md:9`) and explicitly relies on adapter conversions for slot, history, session meta, decisions, and active refine rules (`plans/S-08.plan.md:58`, `plans/S-08.plan.md:63`), but its test strategy has only typecheck, all pytest, and browser walkthrough (`plans/S-08.plan.md:88`, `plans/S-08.plan.md:91`). There is no unit fixture for `FullSnapshotResp` -> sandbox view-model, no regression case for empty `currentRecs` requiring `/recs`, and no faithful-refine coverage for the real branch. Fix: add lightweight adapter tests or a checked fixture script for representative snapshots (`blank`, `running with last_recs`, `after eat with empty currentRecs`, `refine activeRules`) plus a browser/network assertion that post-advance calls `/recs` before showing the next meal.

S-08 self-test explicitly runs under `?mock_recommend=1` (`specs/S-08.md:57`), and `/refine` really does bypass `_impl_refine` in mock mode (`chisha/web_api.py:1727-1742`), so S-08 does not have to prove the real refine pipeline. However, `CONTRACTS.md` also says sandbox behavior should match prod except clock/data isolation and flags fake LLM as dangerous in the general sandbox contract. The `?real=1` escape hatch is therefore the right compromise, but it is only mentioned in changelog/response (`plans/S-08.plan.md:105-106`, `plans/S-08.plan.md:135-136`), while Step 2 still says all POSTs use `withMock(path)` by default (`plans/S-08.plan.md:52`) and Step 4 says "mock_recommend=1 always" (`plans/S-08.plan.md:81`). 6. Async / race behavior: the plan identifies interval cleanup and stale sid guards (`plans/S-08.plan.md:96-97`), but it does not specify where the interval lifetime lives. Also treat backend `cancelled_by_rollback` as terminal failure/non-advance, not as `"running"`, because `backendJobToStatus` currently only lists running/done/failed (`plans/S-08.plan.md:66`) while the backend exposes `cancelled_by_rollback` (`plans/S-08.plan.md:41`). Without this, a rollback during eat polling can spin forever.

7. File-scope / acceptance consistency: the plan says "5 个新文件" in Done When while the real affected set is 3 new files plus 2 modified files (`specs/S-08.md:91`, `plans/S-08.plan.md:6-11`). That wording is minor, but in combination with the activeRules contradiction it makes the implementation boundary unclear. Clean the plan before landing so the executor does not accidentally exceed the S-08 file budget or leave unreachable adapter fields.
