### BLOCKING — `_sandbox_ctx` wrap missing in all 3 fixed endpoints

The dep was added to all 3 signatures (validation + gating now active), but none of the 3 handler bodies wraps its data_root calls in `with _sandbox_ctx(_sid):`. This means:

1. **`api_put_profile`** (lines 515-526): calls `_write_profile_preserving_comments(profile)` → inside calls `_profile_path()` → `data_root.profile_path(ROOT)` which reads `current_sandbox_session()` ContextVar. Without `_sandbox_ctx(_sid)`, ContextVar is never set inside the handler, so the write always goes to prod/default path. The dep validates and gates the sid correctly (400/409/404 fire) but for a valid non-default sid the write silently falls through to default bucket.

2. **`api_feedback_session`** (lines 753-772): calls `feedback_store.load_store(ROOT)` directly without `_sandbox_ctx`. Lower severity (read-only) but same structural defect — sandbox bucket is not consulted.

3. **`api_long_term_prefs_refresh`** (lines 802-850): calls `_profile_path()` and `extract_and_save(root=ROOT)` without `_sandbox_ctx`. The prefs are extracted from and written to the prod/default path regardless of sid.

Compare with correctly-implemented endpoints: `api_history` (line 544: `with _sandbox_ctx(_sid): return _impl_history(days)`) — same dep, body wrapped.

The fix for all 3 is identical: wrap the handler body (after auth/validation guards) in `with _sandbox_ctx(_sid):`.

Required fix per endpoint:

```python
# api_put_profile
with _sandbox_ctx(_sid):
    _write_profile_preserving_comments(profile)

# api_feedback_session  
with _sandbox_ctx(_sid):
    data = feedback_store.load_store(ROOT)
    # ... rest of body

# api_long_term_prefs_refresh
# auth guards (localhost, admin token) stay outside ctx
# then:
with _sandbox_ctx(_sid):
    store = feedback_store.load_store(ROOT)
    profile = yaml.safe_load(_profile_path().read_text(...))
    # ... rest of body
```
