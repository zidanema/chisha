# S-01 Plan Review

## Summary
The plan is mostly aligned with `apps/debug-ui` as a new standalone Vite/React/TS app and avoids touching existing `apps/web`, `apps/debug-ui`, or `chisha` source. The demo code is likely to typecheck under the copied `apps/debug-ui` strict flags, and the dependency set matches `apps/debug-ui`. However, the plan has two implementation blockers: port `5175` is currently occupied in this worktree environment, and `tsconfig.node.json` is both declared as a deliverable and explicitly canceled, contradicting the spec.

## Findings

### BLOCKING issues (must fix before implementation)
1. Port `5175` is not currently free, so the proposed `strictPort: true` dev server will fail to start. Evidence: the plan hardcodes `port: 5175` with `strictPort: true` in `apps/sandbox-lab/vite.config.ts` (`plans/S-01.plan.md:70-73`) and lists port conflict as only an easy-to-diagnose risk (`plans/S-01.plan.md:323`); actual check shows `node ... TCP 127.0.0.1:5175 (LISTEN)` from `lsof -nP -iTCP:5175 -sTCP:LISTEN`. Recommended fix: stop the existing listener before implementation/validation, or change the plan/spec to a free fixed port and update `specs/S-01.md:11-12`, `plans/S-01.plan.md:45`, and `plans/S-01.plan.md:72`.

2. `tsconfig.node.json` is ambiguous: the plan's affected-file tree says to create it, then the detailed section says not to create it, while the task spec still requires it. Evidence: plan file tree marks `apps/sandbox-lab/tsconfig.node.json` as new (`plans/S-01.plan.md:12-13`), the detailed section says "取消上面这个,不建该文件" (`plans/S-01.plan.md:89-105`), and the spec requires `tsconfig.json + tsconfig.node.json` (`specs/S-01.md:13`, `specs/S-01.md:35`). Existing `apps/debug-ui` has only `tsconfig.json`, no node config (`apps/debug-ui/tsconfig.json:1-25`; repo file list contains `apps/debug-ui/tsconfig.json` and `apps/debug-ui/vite.config.ts`, but no `apps/debug-ui/tsconfig.node.json`). Recommended fix: make one decision in the plan. If mirroring `debug-ui`, remove `tsconfig.node.json` from the affected-file list and note the spec deviation. If honoring the spec, add a valid node config and the required Node/Vite typing so `vite.config.ts` can typecheck cleanly.

### WARNING issues (should fix, not blocking)
1. The localStorage key does not follow the observed `chisha:` prefix used by `debug-ui`. Evidence: the plan uses `const KEY = "chisha-sbx:tweaks"` (`plans/S-01.plan.md:172`) for both `getItem` and `setItem` (`plans/S-01.plan.md:176`, `plans/S-01.plan.md:201`); `debug-ui` documents `chisha:theme` (`apps/debug-ui/README.md:36-40`), implements `chisha:theme` (`apps/debug-ui/src/hooks/useTheme.ts:13`, `apps/debug-ui/src/hooks/useTheme.ts:26`, `apps/debug-ui/src/hooks/useTheme.ts:42`), and also uses `chisha:tbCollapsed` (`apps/debug-ui/src/App.tsx:65-69`). Recommended fix: use a `chisha:` key, e.g. `chisha:sandboxTweaks`, unless the project intentionally wants a new prefix and documents that exception.

2. The chrome-devtools validation is close but not fully equivalent to the project self-test wording. Evidence: `CLAUDE.md` requires `mcp__chrome-devtools__navigate`, a golden path plus an edge case, console/network checks, and explicit failure disclosure for `apps/web` or `apps/debug-ui` frontend/proxy changes (`CLAUDE.md:73-85`); the Sandbox Lab design brief also says frontend tasks use chrome-devtools-mcp (`docs/design_briefs/2026-05-19-sandbox-lab.md:190-198`). The plan includes evaluate-style checks for CSS variables, theme, font, console, network, and persistence (`plans/S-01.plan.md:306-312`) but does not explicitly name navigation, screenshot/visual framing, or a clear edge case such as localStorage corruption/private-mode failure. Recommended fix: add an explicit mcp sequence: navigate to `http://127.0.0.1:5175/`, exercise default state, all four accents, light/dark, refresh persistence, corrupted `localStorage` fallback, inspect console/network, and capture a screenshot.

3. Validation omits `npm run build`, so Vite bundling and production asset handling are not checked. Evidence: package scripts include `build: "tsc -b && vite build"` (`plans/S-01.plan.md:42-46`) but required validation lists install, dev server, typecheck, chrome-devtools, persistence, and pytest only (`plans/S-01.plan.md:301-317`). Recommended fix: add `cd apps/sandbox-lab && npm run build`; this is especially useful because `vite.config.ts` is outside the copied `tsconfig.json` include (`apps/debug-ui/tsconfig.json:23-24`) and typecheck alone will not validate all bundling behavior.

4. The plan references `CLAUDE.md` as if the existing mandatory frontend self-test already covers `sandbox-lab`, but it currently only names `apps/web` and `apps/debug-ui`. Evidence: plan says chrome-devtools-mcp validation is mandatory via `CLAUDE.md` (`plans/S-01.plan.md:306`), while `CLAUDE.md` scopes the rule to `apps/web/src/**` and `apps/debug-ui/src/**` (`CLAUDE.md:73-75`) and startup commands only cover backend, `apps/web`, and `apps/debug-ui` (`CLAUDE.md:40-59`, `CLAUDE.md:79`). Recommended fix: either phrase the S-01 validation as a Sandbox Lab-specific requirement from the spec/design brief, or update docs in the later S-10 task as planned (`docs/design_briefs/2026-05-19-sandbox-lab.md:214`) and do not imply the current CLAUDE rule already names `apps/sandbox-lab`.

5. The scaffold introduces a `/api` proxy before S-01 uses any API, which is low risk but should be described as future-facing. Evidence: plan creates a proxy to `VITE_API_TARGET || "http://127.0.0.1:8765"` (`plans/S-01.plan.md:74-79`), while S-01 explicitly does not do backend integration (`plans/S-01.plan.md:341-342`; `specs/S-01.md:40-42`). Existing `debug-ui` uses the same proxy pattern (`apps/debug-ui/vite.config.ts:10-15`), and `chisha.debug_server` exposes `/api/*` via `chisha.web_api` (`chisha/debug_server.py:9-13`, `chisha/debug_server.py:45-46`). Recommended fix: keep the proxy if mirroring `debug-ui`, but note that S-01 should not call `/api/*` yet.

6. Existing `docs/CONTRACTS.md` has sandbox invariants that are not violated by S-01, but the broader Sandbox Lab plan must not contradict them later. Evidence: S-01 creates only frontend scaffold/demo files (`plans/S-01.plan.md:6-22`) and explicitly does not touch `chisha/**` (`plans/S-01.plan.md:24-28`), while `docs/CONTRACTS.md` requires sandbox behavior to match production except clock/data-root isolation (`docs/CONTRACTS.md:93-100`) and says debug-ui is read-only (`docs/CONTRACTS.md:114-120`). Recommended fix: no S-01 code change required; carry these invariants into later backend/session tasks, especially reset/disable L1 locking and inspect raw prefs.

### INFO items (FYI, low risk)
1. There is no root workspace package to update; `apps/sandbox-lab` will not be auto-picked up by npm workspaces. Evidence: the repo has app-level package files at `apps/debug-ui/package.json` and `apps/web/package.json` (`apps/debug-ui/package.json:1-24`, `apps/web/package.json:1-28`), and no root `package.json` exists in this worktree; the plan already flags workspace detection as a check (`plans/S-01.plan.md:321-322`). Recommended fix: do not add root workspaces for S-01 unless the project intentionally moves to a monorepo install model.

2. The proposed demo `App.tsx` and `useTweaks.ts` are likely to compile under `debug-ui` strict flags. Evidence: `debug-ui` has `strict`, `noUnusedLocals`, `noUnusedParameters`, `isolatedModules`, and `jsx: react-jsx` enabled (`apps/debug-ui/tsconfig.json:8-21`); the planned `App.tsx` uses its only import and mapped accent variable (`plans/S-01.plan.md:215-249`), and `useTweaks.ts` uses its declared types, constants, state setter, and effect dependencies (`plans/S-01.plan.md:157-209`). Recommended fix: still run `npm run typecheck`; no likely noUnusedLocals/noUnusedParameters issue was found in the shown source.

3. The proposed package dependencies match `apps/debug-ui`; no missing dependency is evident for the S-01 code shown. Evidence: plan dependencies/devDependencies are React, ReactDOM, type packages, plugin-react, TypeScript, and Vite (`plans/S-01.plan.md:48-58`), matching `apps/debug-ui/package.json` (`apps/debug-ui/package.json:13-23`). The demo uses no router, icons, test runner, CSS framework, or API client. Recommended fix: do not add `react-router-dom`, Tailwind, Vitest, or other deps for S-01 unless new code actually imports them.

4. No broken imports or nonexistent project modules were found in the S-01 plan. Evidence: planned source imports only React, ReactDOM, local `./App`, local CSS, and local `./hooks/useTweaks` (`plans/S-01.plan.md:132-136`, `plans/S-01.plan.md:215`); S-01 explicitly defers mock data, sandbox types, and backend integration (`plans/S-01.plan.md:336-342`). The referenced prototype path exists in the repo (`chidiansha-sandbox/project/sandbox-lab/styles.css:1-10`). Recommended fix: keep S-01 self-contained and avoid importing `chisha` Python modules, `apps/debug-ui` TS utilities, or unimplemented `/api/sandbox/sessions/*` clients until later tasks.

5. Regression risk is low because the plan should add a new app directory only. Evidence: the plan states existing app and Python trees are not touched (`plans/S-01.plan.md:24-28`) and lists only new `apps/sandbox-lab` files (`plans/S-01.plan.md:6-22`). Risk by file: new `apps/sandbox-lab/vite.config.ts` low, because it is app-local and mirrors `debug-ui` proxy (`apps/debug-ui/vite.config.ts:4-22`); new app `package.json` low, because there is no root workspace; new app `tsconfig.json` low, because it is app-local; root `package.json` none, because absent; shared tsconfig none, because absent. Recommended fix: keep implementation scoped to `apps/sandbox-lab/**` and generated lockfile inside that app if npm creates one.

6. A simpler alternative exists but would not satisfy the explicit S-01 scope. Evidence: `apps/web` already has sandbox API client code for current single-session sandbox endpoints (`apps/web/src/lib/sandbox.ts:76-90`), and `chisha.web_api` already exposes `/api/sandbox/init`, `/advance`, `/reset`, `/disable`, `/state`, and `/inspect` (`chisha/web_api.py:838-880`). However, the design brief explicitly chooses a new independent `apps/sandbox-lab/` rather than merging into debug-ui (`docs/design_briefs/2026-05-19-sandbox-lab.md:13-16`), and S-01 is defined as a scaffold prerequisite (`specs/S-01.md:1-3`, `specs/S-01.md:21-23`). Recommended fix: proceed with a sub-app if following D-093; otherwise the simpler option is to prototype the theme/accent demo as an `apps/web` route or static HTML, but that changes the product decision.

7. Missing test coverage is mostly frontend build/validation, not pytest. Evidence: S-01 does not touch Python (`plans/S-01.plan.md:24-28`) and the existing pytest sandbox coverage targets backend clock/API behavior (`tests/test_sandbox_clock.py:1-8`, `tests/test_web_api_sandbox.py:1-16`), while the plan already proposes full pytest as a regression check (`plans/S-01.plan.md:316`). There is no Vitest dependency in either app package (`apps/debug-ui/package.json:13-23`, `apps/web/package.json:13-27`). Recommended fix: add `npm run build` plus chrome-devtools checks; do not add pytest for pure scaffold, and only add Vitest if the implementation introduces testable pure TS helpers beyond `useTweaks`.

8. The current backend `/api/sandbox/*` endpoints differ from the later design brief session endpoints, but S-01 does not call them. Evidence: current code defines `/api/sandbox/init` and `/api/sandbox/advance` among current sandbox endpoints (`chisha/web_api.py:838-880`), while the design brief later proposes `/api/sandbox/sessions...` endpoints (`docs/design_briefs/2026-05-19-sandbox-lab.md:115-131`). S-01 explicitly defers backend integration (`specs/S-01.md:40-42`, `plans/S-01.plan.md:341-342`). Recommended fix: avoid adding any API client in S-01; later S-06/S-08 plans must reconcile endpoint shapes against current `web_api.py`.

## Verdict
VERDICT: BLOCKED
1. Port `5175` is currently occupied, so the planned strict Vite dev server will not start until the port conflict is resolved.
2. `tsconfig.node.json` is simultaneously listed as a new file and canceled, while the spec still requires it; the plan must choose one path before implementation.
