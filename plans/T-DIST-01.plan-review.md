# 1. Plan Overall Soundness Review

**Verdict: approve with amendments; do not approve Step 1 as written until the resource-root strategy is narrowed and tested with an actual built wheel.**

- The plan has the right macro shape: resource/root abstraction before CLI entry point, then skill text, skill install, onboarding, packaging, and end-to-end verification. The order at `plans/T-DIST-01.plan.md:61-127` is directionally correct because `chisha doctor` already reports manifest and install/state-root readiness from the runtime root (`chisha/agent_cli.py:643-715`), so packaging root semantics must be settled before user-facing commands are trusted.
- The plan is internally inconsistent: it says "分 6 步" while listing Step 1 through Step 7 (`plans/T-DIST-01.plan.md:59-127`). Patch: rename it to "7 steps" and make Step 6 depend on a built-wheel inspection gate before Step 7.
- Missing Step 0: build inventory and dist inspection. Current `pyproject.toml` only declares Hatchling, `packages = ["chisha"]`, and no script entry point or resource include rules (`pyproject.toml:34-39`). Before moving files, add a one-commit proof step: `uv build`, inspect the wheel contents, install into a temporary home, and assert `chisha doctor` finds `data/manifest.json`. This catches the packaging boundary directly instead of inferring from repo layout.
- Step 1's directory move is not proven necessary. The plan states `mv prompts chisha/prompts; mv profiles chisha/profiles; mv data chisha/data` (`plans/T-DIST-01.plan.md:65`) and calls the alternative a "hack" (`plans/T-DIST-01.plan.md:77`), but Hatchling can include non-package repo directories into a wheel using explicit build config. The cheaper design is: keep source directories where they are, define a single `resource_root()`/`install_root()` abstraction, and use Hatch `force-include` to place `prompts`, `profiles`, `data`, and the generated skill into the installed layout. Hypothesis: this can map external dirs into either package-relative paths or a package-owned resource subtree without changing import names; it needs one wheel-inspection spike.
- If resources are kept outside `chisha/` in the repo but copied into `chisha/` inside the wheel, do not use `importlib.resources.files("chisha").parent` as the production root. That returns the parent of the installed package and is too close to site-packages global layout. Prefer an explicit package resource root such as `importlib.resources.files("chisha") / "_resources"` or `Path(chisha.__file__).parent`, and make dev mode return the repo root. This keeps the read-only bundle under the installed package instead of depending on sibling directories in site-packages.
- Actual import/path blast radius is larger than the plan's "chisha/ 内文件大约 5 处" (`plans/T-DIST-01.plan.md:73`). Concrete hot-path examples include `agent_cli._root()` (`chisha/agent_cli.py:38-40`), `api._default_root()` (`chisha/api.py:93-94`), `recall.__main__` (`chisha/recall.py:1162-1169`), `rerank.SYSTEM_PROMPT_PATH` (`chisha/rerank.py:32-33`), `refine_intent_v2.PROMPT_PATH_V2` (`chisha/refine_intent_v2.py:38-39`), `feedback.PROMPT_PATH` (`rg` result: `chisha/feedback.py:17-18`), `l1_extractor` prompt loading (`rg` result: `chisha/l1_extractor.py:34,201`), `score` default root (`rg` result: `chisha/score.py:1376`), `web_api.ROOT` (`rg` result: `chisha/web_api.py:38`), and `debug_server.ROOT` (`rg` result: `chisha/debug_server.py:32`). Patch: add `chisha/install_root.py` first, then migrate call sites by category: resource reads, data bundle reads, dev/debug only, and state-root compatibility.
- Import-path blast radius if directories move into `chisha/`: Python imports are probably not the main risk because `prompts`, `profiles`, and `data` are data directories, not modules. The real blast radius is path semantics: `manifest.manifest_path(root)` requires `root / "data" / "manifest.json"` (`chisha/manifest.py:59-60`), `recall.load_zone_data` reads `root / "data" / zone` (`rg` result: `chisha/recall.py:53,60-61`), and `methodology` expects `profiles/methodologies` under the supplied root (`chisha/methodology.py:29-30`). A move flips the canonical root from repo root to package dir and forces every test/helper that constructs a fake root to mirror the new resource layout.
- The plan under-specifies package resource APIs. It says "`include = [...]`" under `[tool.hatch.build.targets.wheel]` (`plans/T-DIST-01.plan.md:113-118`), but current Hatch config has only `packages = ["chisha"]` (`pyproject.toml:38-39`). Patch: specify exact Hatchling config, including whether resources are shipped as files in `chisha/`, via `force-include`, or via an explicit package subdirectory. Then add a test that opens the built wheel and asserts required files are present.
- The `scripts/` decision is missing from the 7-step plan. Step 2 proposes `chisha migrate-state` wrapping `scripts.migrate_state` (`plans/T-DIST-01.plan.md:81-87`), but current build config only packages `chisha` (`pyproject.toml:38-39`), so `scripts.migrate_state` will not be importable after `uv tool install`. Patch: expose migration through `chisha.state_migrate` and a `chisha.cli` command, and leave dev-only scripts out of the wheel.
- The plan should explicitly preserve D-102 install/state separation. `state_root.resolve()` treats `root == project_root()` as production and falls back to `~/.chisha` (`chisha/state_root.py:10-17,43-53`). Any new `install_root()` must either replace `project_root()` consistently or keep `project_root()` semantics stable for tests like `tests/test_state_root_d102.py:48` from the `rg` result.
- The plan should add a distribution smoke test that runs outside the repo. Current pytest config injects repo root via `pythonpath = ["."]` (`pyproject.toml:25-32`), which masks missing wheel data and missing entry points. Patch: add a subprocess test or manual acceptance step that runs with a temp `HOME`, no repo on `PYTHONPATH`, and the built wheel installed.
- Cheaper/stabler path: do Step 1 as "root abstraction + wheel resource inclusion + no directory move" first. Only move directories if wheel inspection or runtime APIs prove Hatch inclusion is too awkward. This reduces churn in high-risk files flagged by `CLAUDE.md:66-72` and keeps tests that copy `real_root / "prompts"` and `real_root / "profiles"` (`rg` result: `tests/test_acceptance_d076_d077.py:40-56`) from being rewritten purely for layout.

# 3. Risks the Author Missed

1. **tests/ fixture refactor effort**
   - Severity: P1
   - Author estimate if given: 0.5-1d for Step 1 resource migration + install_root + fixture changes (`plans/T-DIST-01.plan.md:133`).
   - Your estimate: underestimate; 1-1.5d if directories move, 0.5-1d if using a resource-root abstraction without moving directories.
   - Patch suggestion: before editing tests, run a targeted inventory and split fixtures into `state_root` fixtures and `install_resource_root` fixtures. Anchor points: tests copy real resources from repo paths (`rg` result: `tests/test_acceptance_d076_d077.py:40-56`), assert `state_root.project_root()` equals repo root (`rg` result: `tests/test_state_root_d102.py:48`), and read `pyproject.toml` from repo root (`tests/test_manifest_d102.py:33-40`). A directory move turns many of these into semantic fixture decisions, not mechanical path replacements.

2. **scripts/ directory handling**
   - Severity: P1
   - Author estimate if given: not separately estimated; Step 2 casually wraps `scripts.migrate_state` (`plans/T-DIST-01.plan.md:81-87`).
   - Your estimate: underestimate; 0.5d to classify and move/wrap correctly.
   - Patch suggestion: do not package all `scripts/`. Move only user-facing migration into `chisha.cli` backed by `chisha.state_migrate` (`chisha/state_migrate.py:87-180`), because current `scripts/migrate_state.py` depends on `state_root.project_root()` as its source (`scripts/migrate_state.py:26-34`) and `scripts/` is absent from the Hatch package list (`pyproject.toml:38-39`). Keep collector/tagging/debug scripts as dev tools; their hardcoded repo roots and prompt/data paths (`scripts/tag_via_api.py:39-40`, `scripts/build_manifest.py:63-64`, `chisha/loader.py:579-588`) should not become runtime dependencies of `uv tool install`.

3. **Multi-zone data extensibility**
   - Severity: P1
   - Author estimate if given: deferred as future zone bundle split (`plans/T-DIST-01.plan.md:153-157`).
   - Your estimate: about right for sprint scope, but the manifest/onboard contract needs a small patch now.
   - Patch suggestion: make onboarding read available zones from the installed manifest and actual directories, and fail if they disagree. The current manifest lists both `home` and `shenzhen-bay` (`data/manifest.json:11-14`) while U2 proposes shipping only Shenzhen Bay (`plans/T-DIST-01.plan.md:52`). `scripts/build_manifest.py` discovers zones by scanning `data/` (`scripts/build_manifest.py:29-36`), so a release step must rebuild the manifest after any zone pruning.

4. **Version triple consistency: `chisha.__version__` / `pyproject.toml` / `engine_version` in manifest**
   - Severity: P1
   - Author estimate if given: not listed.
   - Your estimate: about right if adding a release gate; current unit coverage is partial.
   - Patch suggestion: keep the existing guard tying `pyproject.toml` version to `manifest.ENGINE_VERSION` (`tests/test_manifest_d102.py:33-40`, `chisha/manifest.py:26`) and add a built-artifact guard: after wheel install, `chisha doctor` must report `engine_version` equal to package metadata and `data/manifest.json.min_engine_version` must be `<= engine_version` (`chisha/agent_cli.py:703-715`, `data/manifest.json:5`). This catches stale wheels or stale generated manifests that source-tree tests can miss.

5. **`package dir == production` signal in `state_root.py:15` comment**
   - Severity: P1
   - Author estimate if given: not listed.
   - Your estimate: underestimate because this becomes a semantic compatibility point during packaging.
   - Patch suggestion: update the comment and implementation together. Today `project_root()` returns `Path(__file__).resolve().parent.parent` (`chisha/state_root.py:25-27`) and the docstring says production passes that package/repo directory down as root (`chisha/state_root.py:10-17`). If install resources move under `chisha/`, `project_root()` and `install_root()` may diverge; either make `state_root.project_root()` delegate to the new install-root source, or keep it as the legacy repo/package-root signal and introduce a separate `resource_root()` for read-only data. Do not let "root == package dir" silently mean different directories in dev and wheel.
