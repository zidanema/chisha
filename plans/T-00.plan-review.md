# Summary

The plan is directionally correct on the two deliverables: a trace schema bump is required by the trace contract, and `RefineIntent.schema_version` is a low-risk additive dataclass field. However, the plan misses one real read-path dependency: `trace_store.list_traces()` performs its own strict `__version` check and does not call `read_trace()`. Updating only `read_trace()` leaves v1 production traces readable by detail/What-if but invisible in the debug session list.

The plan also needs to account for the D-079 golden fixture because it snapshots `__version`. That is an expected-test-update issue, not a production behavior blocker, but the current plan says all D-079 PR-1 tests must stay green without mentioning the fixture update.

# Deliverable 1 Analysis

`chisha/trace_store.py` currently has `TRACE_SCHEMA_VERSION = 1` and strict equality checks in both `read_trace()` and `list_traces()` (`chisha/trace_store.py:27`, `chisha/trace_store.py:139-142`, `chisha/trace_store.py:186-189`). The plan covers `read_trace()` v1->v2 migration (`plans/T-00.plan.md:58-74`) but does not cover the separate `list_traces()` gate.

`write_trace()` force-sets `trace["__version"] = TRACE_SCHEMA_VERSION` before serialization (`chisha/trace_store.py:68-70`). After a bump, new writes will become v2 automatically. This supports the plan's "migrate on read, do not write back" direction for pure reads, but any caller that reads a v1 trace and then calls `write_trace()` will persist it as v2. That is already how refine merge works (`chisha/web_api.py:160`, `chisha/web_api.py:184-189`) and is acceptable if documented as "not automatic on read."

The proposed helper `append_hard_filter_event(trace, event)` is compatible with the existing trace shape because production trace assembly stores L1 under `trace["l1"]` from `_build_l1_trace()` (`chisha/api.py:367-370`, `chisha/api.py:472-490`). What-if reuses base L1 as-is in response traces (`chisha/debug_what_if.py:358-383`), so adding `l1.hard_filter_events` to the normalized read shape will carry through to What-if without requiring L2/L3 recomputation changes.

# Deliverable 2 Analysis

`RefineIntent` is a dataclass with defaulted fields and `to_log_dict()` delegates to `asdict()` (`chisha/refine_intent.py:55-78`). Adding `schema_version: str = "1.0"` as a defaulted field is backward compatible for current direct constructors in tests and scoring code (`tests/test_intent_score.py:125`, `tests/test_intent_score.py:314`, `tests/test_score_v2.py:337-344`).

`is_empty()` only considers semantic intent fields and does not inspect `raw_text` or `freeform_note` (`chisha/refine_intent.py:80-88`). The new schema field should likewise not be included in `is_empty()`; otherwise every empty intent would become non-empty.

`parse_refine_intent("")` returns `RefineIntent(raw_text="")` (`chisha/refine_intent.py:352-354`), and non-empty parse returns a `RefineIntent(...)` with explicit semantic fields (`chisha/refine_intent.py:368-384`). Both paths will get the default schema version if the field is added to the dataclass.

# Missed Dependencies

1. `trace_store.list_traces()` is a missed v1 compatibility path. It reads JSON directly and skips any trace whose `__version` is not exactly `TRACE_SCHEMA_VERSION` (`chisha/trace_store.py:179-189`). The plan only changes `read_trace()` (`plans/T-00.plan.md:58-74`). Result: after `TRACE_SCHEMA_VERSION = 2`, existing v1 traces can still be opened by `GET /api/debug/sessions/{sid}` if `read_trace()` is updated, but they disappear from `GET /api/debug/sessions` because `web_api.api_debug_sessions()` calls `list_traces()` (`chisha/web_api.py:904-909`).

2. The D-079 golden fixture snapshots `__version`. `_normalize_for_golden()` includes `"__version": trace.get("__version")` (`tests/test_d079_pr1_golden.py:108-114`), and the checked-in fixture currently stores `"__version": 1` (`tests/fixtures/recommend_golden.json:104`). Bumping `TRACE_SCHEMA_VERSION` to 2 will fail `test_recommend_meal_golden_snapshot` unless the plan updates or deliberately regenerates that fixture (`tests/test_d079_pr1_golden.py:181-188`).

3. Frontend backend type definitions are permissive enough for the version bump but do not declare `l1.hard_filter_events`. `BackendDebugTrace.__version` is just `number` (`apps/debug-ui/src/api/backend-types.ts:314-316`), so v2 is fine. `BackendL1Recall` currently has a fixed declared field set and no index signature (`apps/debug-ui/src/api/backend-types.ts:13-48`), but TypeScript structural typing does not reject extra runtime fields. This is not a blocker for T-00, but future UI consumption of the event log should update the type.

# Broken Assumptions

1. "read_trace migration covers v1 read compatibility" is incomplete because `list_traces()` is also a read path and bypasses `read_trace()` (`chisha/trace_store.py:179-189`). This is the main broken assumption.

2. The "migrate on read, don't write back" pattern works for pure detail reads and What-if. `api_debug_session_detail()` returns the `read_trace()` result without persisting it (`chisha/web_api.py:923-956`), and `what_if_rerun()` reads the base trace then returns a preview without `write_trace()` (`chisha/debug_what_if.py:161-170`, `chisha/debug_what_if.py:265-280`). `_build_what_if_trace()` copies base L1 into the preview (`chisha/debug_what_if.py:358-383`), so injected `l1.hard_filter_events = []` will be visible in the preview and will not disturb L2/L3 behavior.

3. The pattern is not "never write back" for refine merge. `/api/refine` reads the base trace, mutates `refine` and `__config`, then calls `write_trace()` (`chisha/web_api.py:160`, `chisha/web_api.py:184-189`). Because `write_trace()` force-sets current version (`chisha/trace_store.py:68-70`), a v1 trace refined after the bump will be upgraded on disk. That is consistent with "next write lands v2" in the plan (`plans/T-00.plan.md:74`), but the wording should make clear this applies to any later write, including refine persistence.

4. `test_d079_pr1_frozen_injection` does not rely on the trace dict shape. It tests `rank_combos`, `fallback_rerank`, and `_pick_explore` injection boundaries directly (`tests/test_d079_pr1_frozen_injection.py:17-202`). Adding `l1.hard_filter_events` does not affect those tests.

# Regression Risk

1. `baseline_l2_snapshot.py` does not compare runtime trace envelope bytes. It builds compact L2 snapshots from `recall()`, `rank_combos()`, and `apply_caps()` (`scripts/baseline_l2_snapshot.py:45-77`) and writes only `snap_*.json` files (`scripts/baseline_l2_snapshot.py:102-107`). `compare_traces.py` compares metadata, combo order, scores, and breakdown keys (`scripts/compare_traces.py:63-81`). Adding `l1.hard_filter_events: []` to runtime trace files will not affect this baseline tool unless L1/L2 business logic changes.

2. The actual byte/fixture regression risk is the D-079 golden fixture, not `baseline_l2_snapshot`. The golden test includes trace `__version` (`tests/test_d079_pr1_golden.py:108-114`) and will see 1->2 as a snapshot diff against `tests/fixtures/recommend_golden.json:104`.

3. Adding an empty `hard_filter_events` list to new trace writes should not break What-if equivalence assertions. `test_what_if_zero_overrides_matches_replay` compares final IDs, L2 top scores/breakdowns, L2 dim stats, and source markers (`tests/test_d079_pr2_what_if.py:102-169`), not full trace equality. `test_what_if_overrides_n_return` asserts `wi["__frozen"] == base_trace["__frozen"]` (`tests/test_d079_pr2_what_if.py:276-289`), and the new field is under `l1`, not `__frozen`.

# Simpler Alternatives

Keeping `TRACE_SCHEMA_VERSION = 1` and treating `hard_filter_events` as additive would reduce compatibility churn, but it conflicts with the explicit contract: "改 trace schema 必 bump `TRACE_SCHEMA_VERSION`" (`docs/CONTRACTS.md:104-107`). The plan's v2 bump is therefore correct.

The simpler safe implementation is not "no bump"; it is "bump to v2 and centralize accepted-version normalization." At minimum, both `read_trace()` and `list_traces()` need the same accepted-version logic because both currently inspect raw `__version` independently (`chisha/trace_store.py:139-142`, `chisha/trace_store.py:186-189`).

# Missing Test Coverage

1. Add a v1 compatibility test for `list_traces()`, not only `read_trace()`. Current tests cover strict list behavior for current-version traces and corrupt files (`tests/test_trace_store.py:109-137`), but none cover listing an older accepted schema version. This guards the `GET /api/debug/sessions` Sidebar path (`chisha/web_api.py:904-909`).

2. Add an endpoint-level v1 compatibility test for `GET /api/debug/sessions` and/or `GET /api/debug/sessions/{sid}`. Existing endpoint tests cover current-version detail, unknown-version 409, and legacy no-`__frozen` with current version (`tests/test_d079_pr2_endpoints.py:158-198`, `tests/test_d079_pr2_endpoints.py:266-310`), but not `__version == 1` after the app's current version becomes 2.

3. Preserve the D-079 What-if invariants already covered: zero overrides must match Replay L2/final (`tests/test_d079_pr2_what_if.py:102-169`), What-if must use frozen L1 prefs (`tests/test_d079_pr2_what_if.py:174-189`), frozen today (`tests/test_d079_pr2_what_if.py:192-212`), frozen meal log (`tests/test_d079_pr2_what_if.py:215-272`), and frozen prefs in the LLM path (`tests/test_d079_pr2_what_if.py:340-452`). No new hard-filter-event test currently asserts those invariants remain unchanged; running the existing D-079 PR-2 suite is necessary.

4. Add a `RefineIntent.schema_version` parser test as planned (`plans/T-00.plan.md:96-97`). Also assert `parse_refine_intent("").is_empty()` remains true after adding the field, because `is_empty()` is used to decide whether refine intent is injected into context and trace (`chisha/refine.py:95-101`, `chisha/refine_intent.py:80-88`).

VERDICT: BLOCKED 1. `list_traces()` v1 compatibility is missing, so old v1 traces disappear from `/api/debug/sessions` after the bump. 2. D-079 golden fixture/test update for `__version` is not accounted for.
