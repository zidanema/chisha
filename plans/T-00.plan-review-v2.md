## Blocker 1 (prior) В· `list_traces()` missing v1 compatibility

**Original issue**: `list_traces` was doing strict `__version == 1` rejection; v1 traces would never appear in Sidebar.

**Revised plan says** (Step 3, lines 60вҖ“88):

> `ACCEPTED_TRACE_VERSIONS = {1, 2}`
>
> `def list_traces(...):`  
> `    version = data.get("__version")`  
> `    if version not in ACCEPTED_TRACE_VERSIONS:`  
> `        continue  # дёҚз®— corrupt, йқҷй»ҳи·іиҝҮ (зҺ°жңүиҜӯд№үдҝқз•ҷ)`  
> `    # list и·Ҝеҫ„дёҚйңҖиҰҒжіЁ hard_filter_events (Sidebar еҸӘиҜ» top1_summary зӯүе…ғдҝЎжҒҜ)`

The plan defines `_normalize_to_current_version` and shows it called inside `read_trace`. `list_traces` intentionally **skips** the normalizer вҖ” the plan explicitly justifies this: list returns metadata only (top1_summary etc.), not full trace payloads, so injecting `hard_filter_events` is unnecessary.

The original blocker was v1 traces being excluded from the list. That is now fixed: both functions use `ACCEPTED_TRACE_VERSIONS = {1, 2}`, so v1 traces appear in Sidebar. The deliberate separation of normalization (read-only) from listing is sound and documented.

**STATUS: RESOLVED**

## Non-Blocking Notes

1. `list_traces` silently skips traces with unknown versions (`continue`) вҖ” this means future version bumps to v=3 would silently drop traces from the list without any warning log. Minor: add a `logger.warning` on skip is a nice-to-have but not a blocker.
2. `_normalize_to_current_version` does not update `__version` in memory (intentionally preserves disk record). This is documented but means callers cannot rely on `data["__version"] == 2` after normalization вҖ” acceptable given the design, just worth a comment in code.
