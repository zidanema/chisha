## 现在能用什么

个人项目，自用打磨中，核心链路已跑通（详见 [README](../README.md)）：

- 三阶段推荐：L1 召回（硬过滤 + 软偏好）→ L2 15 维打分 → L3 LLM 精排写理由
- Web SPA 用户视图 + 反馈系统（差评下一次推荐就降权/剔除，好评优先）
- 长期口味画像（从真实反馈统计聚合，可解释、可手编）
- 多轮 refine（自然语言追加约束）
- 一行安装 + Claude Code 自动触发 skill，状态独立存在 `~/.chisha/`

**当前边界**：仅深圳湾办公区（334 家 / 22556 菜）+ 哈佛餐盘方法论 + Claude Code 自动接入（其它宿主可手动调 CLI）。

## 接下来的方向

按"真有人要"拉动，不预先排期。优先级：自用稳定 > 同事推广 > 开源。

**让别人也能用（推广前要补的）**
- **多工区数据包**：把数据按办公区拆成可独立订阅的包（深圳科技园 / 北京中关村 / 上海…），不用装别人城市的数据
- **可插拔方法论**：哈佛餐盘之外（增肌 / 糖控 / 孕期 / 低 FODMAP…）做成可配置模板，而不是硬编码
- **多宿主适配**：Claude Code 之外（Cursor / Codex / OpenClaw…）也能自动触发，不只手动调 CLI

**体验增强（想做但不急）**
- 到点自动提醒 / 推送到 IM（当前是用户主动问）
- 探索机制更聪明（最近没吃过的菜系/食材主动冒出来）

**数据链路（自用稳定后重构）**
- 把采集 / 清洗打标 / 对外数据服务拆成清晰的子模块，推荐层只消费数据服务、不读硬路径

## 已砍清单（明确不做）

> 这些功能讨论过、**已经决定不做**。想推翻先去 [decisions.md](decisions.md) 加一条新决策说明为什么。

### 产品形态

| 功能 | 为什么砍 |
|---|---|
| SaaS 平台 / 用户系统 / 登录计费 | 运营成本高、隐私顾虑、单用户协同无意义 |
| 独立公网 APP / 小程序 | 定位是接进 AI 助手的本地工具，不是又一个 APP |
| MCP Server 包装 | 走 CLI + Skill 模式，更贴近宿主 agent 的使用习惯 |

### 功能边界

| 功能 | 为什么砍 |
|---|---|
| 卡路里精确追踪 | 太重，违背"轻量决策"定位 |
| 在家做饭推荐 / 食材采购建议 | 数据形态不同，不在外卖场景内 |
| 体重 / 体脂记录 | 已有大量专业 APP |
| 训练计划生成 | 是另一个产品 |
| 商家平台直接下单 | 各 APP 接口封闭，跳转就够 |
| 社交分享（晒午餐） | 偏离工具属性 |

### 推荐逻辑

| 想法 | 为什么砍 |
|---|---|
| 严格 1/2-1/4-1/4 餐盘硬约束 | 中式外卖现实下不可达，召回会被卡死，改弱约束 |
| 把全量候选丢给 LLM 挑 | token 爆炸 + 大候选下约束满足效果差，改"召回打分 → LLM 精排 top N" |
| 让 LLM 在上百候选里挑选排序 | LLM 在挑选/排序上引入随机性，靠打分排序更稳 |
| 个人偏好按"店::菜"粒度记忆 | 样本太稀疏、信号弱，改（菜系, 烹饪方式, 主料）粒度 |
| LLM 蒸馏自然语言"口味洞察" | N=1 数据下过拟合假规律，改统计聚合 |
| 本仓做数据采集 | 采集由 sister project `chisha-collector` 独立维护 |
| 训练日感知（练后加蛋白） | 增益小、复杂度高 |
| 全局协同过滤（"和你类似的人爱吃"） | 单用户场景没意义 |
| 让用户主动选 mood 标签 | 增加摩擦 + 误导信号，改结构化意图解析 |
| 定时服务自动拉起 | 自用阶段手动开页面没毛病，不是技术债 |

### 反馈交互

| 想法 | 为什么砍 |
|---|---|
| 单一"满意度"星级 | 好吃和满意是两个维度，拆开 |
| 纯文本作为唯一反馈方式 | 摩擦太大，多数人不填 |
| 反馈提交后可改 | 下游学习不稳定，改 append-only |
