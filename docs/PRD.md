# 今天吃点啥 · 产品需求文档（PRD）

> 这份文档讲清楚**为什么做、做给谁、做成什么样**。
> 这是个人项目：我自己每天在公司纠结点啥外卖、又想认真减脂增肌，于是做了它，现在开源出来。
>
> 项目名：今天吃点啥 · 代码名：`chisha`

---

## 1. 一句话定位

**「今天吃点啥」（`chisha`）= 原则派点餐助手——给已经认定一套吃法、但懒得每天选店的原则派，30 秒搞定外卖决策。**

这里有个关键区分：产品定位是**"已有方法论用户的执行外包"**，不是"健康结构推荐器"。我不负责帮你*想清楚*该怎么吃（那是你自己认定的方法论），我负责把你认定的吃法每天*落地*。"控油 + 有蔬菜 + 有蛋白"只是我（哈佛餐盘）这一套方法论的具体落地，它描述的是"我"的吃法，没说清产品服务的是**什么人**——服务的是**原则派**：已经认定一套饮食原则、缺的只是每天落地的执行力。

---

## 2. 缘起

### 2.1 我在解决自己的什么问题

我每天中午和晚上点外卖，**至少花 10-20 分钟在点评、美团里翻**。今天吃啥？哪家不踩雷？哪家口味合适？吃了一周湘菜要不要换换？这些问题每天问两遍，决策疲劳真实存在。

更糟的是，我同时在做减脂增肌。早期试过几种路线：

- **算热量**：太重，每口都要查，坚持不下去
- **吃干净**：太苦，完全戒油戒辣，社交场合很尴尬
- **哈佛餐盘比例**（1/2 蔬菜 + 1/4 蛋白 + 1/4 主食）：相对宽松，逻辑简单，但**每顿点餐时手动凑齐这个比例还是费劲**

第三种方法论我认了，但落地依然累——每次点餐还得自己想：这家有蔬菜吗？这道菜油重不重？要不要再加一份青菜？这些判断其实有规律，但每天重复做就是消耗。

**有了一份周边商家 + 商品的数据后，我意识到这个问题可以系统性解决**：让工具在每顿饭时自动给我推一组组合，每个都已经按结构配好，我只要从里面选一个。决策从"翻 30 分钟"变成"扫 30 秒"。

### 2.2 餐盘策略：弱约束而不是硬比例（重要）

中式外卖的现实是：

- 单道盖饭/粉面：天然达不到 50% 蔬菜
- 拼 2-3 道菜：客单 60-100+，吃不完会剩，但可接受

我不要求每顿严格 1/2-1/4-1/4。**我接受推荐里菜量略多，自己在吃的时候控制比例（多吃菜、不全吃肉）**。工具的任务是保证三件事：

1. **控油**：避开油炸、油焖、爆炒重油
2. **有蔬菜**：每顿至少包含 1 道明确的蔬菜（叶菜/瓜菜/菌菇）
3. **有蛋白**：每顿明确有 1 份动物蛋白或豆制品

至于"刚好 1/2 比例"——靠人脑在饭桌上调，不靠算法在召回时凑。这是刻意的设计取舍：在中式外卖现实下，硬卡比例会把召回卡死；结构正确比绝对精确重要。客单价 60-100 元、偶尔剩饭，都是可接受代价。

### 2.3 为什么不直接用 Keep / 薄荷健康 / 美团的 AI 助手

这些工具我都看过，痛点有三个：

1. **数据是死的，不懂我**：它们不知道我"喜欢带汤水的炖煮，不爱重酱汁的红烧"这种细节
2. **没有反馈闭环**：吃完了好不好？分量够不够？太油了？没人接住，下次还推同款
3. **绑死在自家生态**：我希望它跟通用 Agent（Claude Code / 任何能跑 CLI 的 Agent）打通，能在我已经在用的终端里出现，而不是又装一个 APP

所以我决定**自己做一个**，做成 Agent 友好的开源工件，能进我的工作流。

## 3. 服务谁

### 3.1 主用户：我自己

- 深圳湾办公区上班（数据 zone = `shenzhen-bay`）
- 中午晚上都点外卖为主
- 已经在用长程 Agent（Claude Code 等）
- 体重控制 + 力量训练期，认弱餐盘方法论（控油 + 有蔬菜 + 有蛋白），额外约束：当顿饱腹感必须够（轻食/小份子餐撑不住，得补一道硬菜）
- 写代码出身，不怕命令行，喜欢可控的工具

### 3.2 扩展用户：公司同事

- 字节多个城市办公区（深圳科技园、北京中关村、上海等）
- 每个工区一份独立的数据子包，按工区订阅
- 自己也在用长程 Agent，能跑 Python / 接 CLI
- 各自有不同的口味和饮食目标，但方法论代码可以共享

### 3.3 远期用户：开发者社区

- 各地的开发者，可能用不同 LLM、不同 Agent、不同终端
- 自己采集本地数据（用 sister project `chisha-collector`），复用 `chisha` 推荐方法论
- 可能有人改造 fork，做出自己城市的版本

### 3.4 不服务的人

我想清楚地说：

- **目标缺失型用户**——"什么都行又什么都不想吃" / 完全不在乎吃啥。这类用户的核心问题是**目标缺失**，需要的是娱乐化推荐或目标推导，不是执行外包。同一产品很难同时服务"目标缺失"和"执行摩擦"两种问题，明确不做。
- **不会做饭的人**——这个工具只解决"点啥外卖"，不解决"在家做啥"
- **严格控卡党**——我不做精确卡路里追踪，方法论是"结构正确"不是"绝对低脂"
- **要严格 1/2-1/4-1/4 餐盘比例的人**——见 §2.2，我做的是弱约束
- **不愿意接 Agent 的人**——这是个接进 AI 助手的工具不是独立 APP，没 Agent 用不了
- **完全没自己数据的人**——数据采集是有门槛的，没数据推不了

## 4. 核心痛点（按优先级）

| 优先级 | 痛点 | 解法 |
|---|---|---|
| **P0** | 每顿饭花 10-20 分钟决策点啥 | 一次给一组候选（当前 5 个），30 秒选一个 |
| **P0** | 想吃健康但每次手动凑"控油+有菜+有蛋白"累 | 召回阶段自动满足三件套，打分时偏好低油 |
| **P1** | 同样的菜，A 店做得好 B 店做得差，没法区分 | 反馈系统按"菜系×烹饪方式×主料"粒度记忆个人化偏好 |
| **P1** | 吃来吃去就那几家，新店发现不到 | 探索机制每组留 1-2 个候选给"没吃过的" |
| **P2** | 减脂期容易吃得太重口/太油 | 召回 + 打分双重控油，软约束不强制 |
| **P2** | 有时候想清淡有时候想解馋，没法表达"今天就想吃辣" | refine 多轮收敛，自然语言追加约束 |

## 5. 用户故事（典型场景）

### 故事 1：工作日中午（最高频）—— 已实现

> 11:25，我在写代码。我跟 Claude Code 说一句"中午吃啥"，它调起 chisha，在本机 Web 页面弹出 5 个组合（3 个稳妥 + 2 个我没吃过的探索）：
>
> - **潮汕牛肉粿条 + 蒜蓉空心菜**（高蛋白低油，本周还没吃过潮汕菜）
> - **湘里湘亲：清炒胡萝卜肉丝 + 油麦菜 + 米饭**（清炒控油）
> - **来盖一份饭：茄子烧豆腐盖饭 + 一份青菜**（豆制品换换口味）
> - …（另外两个）
>
> 我扫一眼，第二个看着不错，点选它，跳到点评 APP 对应商家页下单。

**关键体验点**：从开口到下单跳转，不超过 1 分钟。当前是**我主动问**（在 Claude Code 里说"中午吃啥"触发）；"到点自动提醒 / 推送到 IM"是后续设想，还没做。

### 故事 2：饭后反馈 —— 已实现

> 14:00，我刚吃完饭回工位。我给中午那餐打个反馈：
>
> "中午的清炒胡萝卜肉丝 + 油麦菜 + 米饭怎么样？
>
> 好吃度: ⭐⭐⭐⭐⭐
> 整体满意: ⭐⭐⭐⭐⭐
> [偏油] [分量小] [配送慢] [想再来]
> 备注：（可选）"
>
> 我点 4 星，勾选"想再来"。下次这家就会优先被推；要是给了差评，下一次推荐它就会被降权甚至剔除。

**关键体验点**：几秒内完成，不破坏工作节奏；反馈下一次推荐就生效。

### 故事 3：临时调整（refine）—— 已实现

> 18:00，今天加班晚下班，想吃顿好的。
>
> 我对 Agent 说："今天想吃辣的、解馋点。"
>
> chisha 重新推一组，全是中辣以上的下饭菜。

**关键体验点**：自然语言追加约束，不需要点选项。

### 故事 4：长期偏好进入推荐 —— 已实现

> 用了两周后某天，推荐的组合里有好几个都是带汤水的炖煮（潮汕牛肉、酸菜鱼）。
>
> 我去看个人画像，确实**统计聚合**显示我对"潮汕煮""川菜炖"采纳率最高、平均评分最高。这是数得出来的偏好在起作用——不是 LLM 编出来的"洞察"，而是从我的真实反馈里统计出来的。

**关键体验点**：用得越久越懂你，且**懂的逻辑可解释、可手编**——不是黑盒蒸馏，避免 N=1 数据下过拟合假规律。

### 故事 5：同事接入 —— 规划中

> 同事 B 在北京中关村办公区。看我用得不错，问要怎么用。
>
> 我告诉他（这是规划中的同事接入形态，当前只有深圳湾 + Claude Code）：
> 1. 拿一份自包含 skill bundle（形态B：代码 + 北京中关村数据包 + 依赖在一个文件夹，按工区拆包，他不用装深圳数据），拷进 `~/.claude/skills/chisha/`
> 2. 跑 `onboard` 初始化，bundle 内 wrapper 直接被他的 Agent 驱动（无需全局装包）
> 3. 写自己的 `profile.yaml`
>
> 他用的是别的模型不是 Claude，数据存在他自己机器上，反馈也只他自己看得到。

**关键体验点**：本地闭环，数据隐私彻底；按工区拆包，按需订阅。（跨工区数据包 + 多 host 自动接入仍在规划，见 [ROADMAP.md](ROADMAP.md)。）

## 6. 北极星指标

**唯一最重要的指标：连续使用情况。**

| 指标 | 自用 1 周目标 | 自用 1 月目标 |
|---|---|---|
| 7 日连续工作日采纳率（一组里选 1 的比例） | ≥ 50% | ≥ 60% |
| 餐后好吃度 ≥ 4 星比例 | — | ≥ 70% |
| 周内主动开启推荐次数 | ≥ 8 次/周 | ≥ 10 次/周（中晚 × 工作日 5） |

**为什么不用"决策时间从 15min → 1min"**：这个指标不可度量、易作弊（用户自己改流程不算产品功劳）。所以改用连续采纳率。

**不优先的指标**：

- 用户体重变化、卡路里数字（受太多因素影响，不可控）
- 营养素精确度（方法论是"结构正确"不是"绝对精确"）
- LLM API 成本（单次推荐成本极低，可忽略；何况智能由宿主 agent 提供，chisha 自己不持 key）
- 推荐延迟（5s 还是 30s 不影响"省决策"的体验）

## 7. 不做的事（明确边界）

避免功能蔓延：

| 不做 | 为什么不做 |
|---|---|
| 卡路里精确追踪 | 太重，违背"轻量决策"定位 |
| 严格 1/2-1/4-1/4 餐盘硬约束 | 中式外卖现实下不可达，改弱约束（见 §2.2） |
| 在家做饭推荐 | 数据形态完全不同，先聚焦外卖场景 |
| 食材采购建议 | 同上 |
| 体重 / 体脂记录 | 已有大量专业 APP，不重复造轮子 |
| 训练计划生成 | 是另一个产品，可以未来独立做 |
| 商家平台对接（直接下单） | 各 APP 接口封闭，靠跳转就行 |
| 社交分享（"晒午餐"） | 偏离工具属性 |
| 独立公网 APP / SaaS 多租户 | 运营成本高、隐私顾虑、单用户协同无意义；本工具是本地单用户 |
| 数据采集（爬取/清洗/打标） | 由 sister project `chisha-collector` 负责，本仓只消费数据（见 §9） |
| LLM 蒸馏自然语言"洞察" | N=1 数据下容易过拟合假规律，改统计聚合 |

## 8. 成功是什么样

### 自用阶段成功

- 自己每天用 ≥ 1 顿
- 7 日工作日连续采纳率 ≥ 50%
- 我愿意拍着胸脯把它推荐给同事

### 同事推广阶段成功

- 至少 3 个同事在自己 Agent 里跑起来（覆盖至少 2 个工区）
- 反馈系统进入闭环，统计聚合的偏好画像让推荐越来越准
- 出现至少一个"我没想到它居然推得这么准"的瞬间

### 开源阶段成功

- 至少几个外部 PR（功能贡献或数据贡献）
- 至少 1 个其他城市的数据 fork
- 出现一个我没接入过的 Agent 形态用户

## 9. 数据来源边界（必读）

本项目**不负责数据采集**。数据采集由 sister project `chisha-collector` 维护：

- 采集源、清洗、打标、保鲜由 collector 负责
- 本仓 `chisha` 只消费 collector 输出的 `restaurants.json` + `dishes_tagged.json`
- 数据 schema 看代码 `chisha/loader.py` + `data/{zone}/restaurants.json` 实际结构（不在文档里维护一份）
- Collector 与 chisha 各自独立演进、独立发版
