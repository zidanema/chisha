# 今天吃点啥 · 产品需求文档（PRD）

> 这份文档讲清楚**为什么做、做给谁、做成什么样**。
> 写给和我一样在公司纠结点啥外卖、又想认真减脂增肌的开发者朋友。
>
> 项目名：今天吃点啥 · 代码名：`chisha`

---

## 1. 一句话定位

**「今天吃点啥」（`chisha`）是一个能接进你 Agent 的点餐决策助手，让你不用每顿纠结，又能稳定吃出"控油 + 有蔬菜 + 有蛋白"的健康结构。**

---

## 2. 缘起

### 2.1 我在解决自己的什么问题

我每天中午和晚上点外卖，**至少花 10-20 分钟在点评、美团里翻**。今天吃啥？哪家不踩雷？哪家口味合适？吃了一周湘菜要不要换换？这些问题每天问两遍，决策疲劳真实存在。

更糟的是，我同时在做减脂增肌。早期试过几种路线：

- **算热量**：太重，每口都要查，坚持不下去
- **吃干净**：太苦，完全戒油戒辣，社交场合很尴尬
- **哈佛餐盘比例**（1/2 蔬菜 + 1/4 蛋白 + 1/4 主食）：相对宽松，逻辑简单，但**每顿点餐时手动凑齐这个比例还是费劲**

第三种方法论我认了，但落地依然累——每次点餐还得自己想：这家有蔬菜吗？这道菜油重不重？要不要再加一份青菜？这些判断其实有规律，但每天重复做就是消耗。

**有了一份周边商家+商品的数据后，我意识到这个问题可以系统性解决**：让 Agent 在每顿饭时自动给我推 3 个组合，每个组合都已经按结构配好，我只要从 3 个里选一个。决策从"翻 30 分钟"变成"扫 30 秒"。

### 2.2 餐盘策略：弱约束而不是硬比例（重要）

中式外卖的现实是：

- 单道盖饭/粉面：天然达不到 50% 蔬菜
- 拼 2-3 道菜：客单 60-100+，吃不完会剩，但可接受

我不要求每顿严格 1/2-1/4-1/4。**我接受推荐里菜量略多，自己在吃的时候控制比例（多吃菜、不全吃肉）**。Agent 的任务是保证三件事：

1. **控油**：避开油炸、油焖、爆炒重油
2. **有蔬菜**：每顿至少包含 1 道明确的蔬菜（叶菜/瓜菜/菌菇）
3. **有蛋白**：每顿明确有 1 份动物蛋白或豆制品

至于"刚好 1/2 比例"——靠人脑在饭桌上调，不靠算法在召回时凑。客单价 60-100 元、偶尔剩饭，都是可接受代价。

详见 [DESIGN.md §5.3 plate_rule](../DESIGN.md)、决策 [D-023](DECISIONS.md#d-023)。

### 2.3 为什么不直接用 Keep / 薄荷健康 / 美团的 AI 助手

这些工具我都看过，痛点有三个：

1. **数据是死的，不懂我**：它们不知道我"喜欢带汤水的炖煮，不爱重酱汁的红烧"这种细节
2. **没有反馈闭环**：吃完了好不好？分量够不够？太油了？没人接住，下次还推同款
3. **绑死在自家生态**：我希望它跟我的 Agent（OpenClaw / Claude Code / HappyClaw）打通，能在我的飞书里推卡片，而不是又装一个 APP

所以我决定**自己做一个**，做成 Agent 友好的开源工件，能进我的工作流。

## 3. 服务谁

### 3.1 主用户：我自己（Phase 1）

- 深圳科技园上班
- 中午晚上都点外卖为主
- 已经在用 OpenClaw / HappyClaw 这种长程 Agent
- 体重控制 + 力量训练期，认弱餐盘方法论（控油 + 有蔬菜 + 有蛋白），
  额外约束：当顿饱腹感必须够（朋友A失败教训, 见 [D-044](DECISIONS.md#d-044)）
- 写代码出身，不怕命令行，喜欢可控的工具

### 3.2 扩展用户：公司同事（Phase 2）

- 字节多个城市办公区（深圳科技园、北京中关村、上海等）
- 每个工区一份独立的数据子包，按工区订阅
- 自己也在用长程 Agent，能跑 Python / 接 MCP
- 各自有不同的口味和饮食目标，但方法论代码可以共享

### 3.3 远期用户：开发者社区（Phase 3）

- 各地的开发者，可能用不同 LLM、不同 Agent、不同 IM
- 自己采集本地数据（用 sister project `chisha-collector`），复用 `chisha` 推荐方法论
- 可能有人改造 fork，做出自己城市的版本

### 3.4 不服务的人

我想清楚地说：

- **不会做饭的人**——这个工具只解决"点啥外卖"，不解决"在家做啥"
- **严格控卡党**——我不做精确卡路里追踪，方法论是"结构正确"不是"绝对低脂"
- **要严格 1/2-1/4-1/4 餐盘比例的人**——见 §2.2，我做的是弱约束
- **不愿意接 Agent 的人**——这是个 Skill 不是 APP，没 Agent 用不了
- **完全没自己数据的人**——数据采集是有门槛的，没数据 Skill 推不了

## 4. 核心痛点（按优先级）

| 优先级 | 痛点 | Skill 的解法 |
|---|---|---|
| **P0** | 每顿饭花 10-20 分钟决策点啥 | Agent 主动推 3 个候选，30 秒选一个 |
| **P0** | 想吃健康但每次手动凑"控油+有菜+有蛋白"累 | 召回阶段自动满足三件套，打分时偏好低油 |
| **P1** | 同样的菜，A 店做得好 B 店做得差，没法区分 | 反馈系统按"菜系×烹饪方式×主料"粒度记忆个人化偏好（V2，见 [D-025](DECISIONS.md#d-025)） |
| **P1** | 吃来吃去就那几家，新店发现不到 | 探索机制每次留 1-2 个候选给"没吃过的"（V2.1） |
| **P2** | 减脂期容易吃得太重口/太油 | 召回 + 打分双重控油，软约束不强制 |
| **P2** | 有时候想要清淡有时候想要解馋，没法表达"今天就想吃辣" | refine 多轮收敛，自然语言追加约束（V2.1） |

## 5. 用户故事（典型场景）

### 故事 1：工作日中午（最高频）—— V1 就实现

> 11:25，我在写代码，专注度正高。OpenClaw 弹一个飞书卡片：
>
> "中午给你三选一：
> - **潮汕牛肉粿条 + 蒜蓉空心菜** （高蛋白低油，本周还没吃过潮汕菜）
> - **湘里湘亲：清炒胡萝卜肉丝 + 油麦菜 + 米饭** （清炒控油）
> - **来盖一份饭：茄子烧豆腐盖饭 + 一份青菜** （豆制品换换口味）"
>
> 我扫一眼，第二个看着不错，点选 2。OpenClaw 给我跳到点评 APP 对应商家页。

**关键体验点**：从看到卡片到下单跳转，不超过 1 分钟。**V1 主动推承诺不变**，但形态分两段：[D-049](DECISIONS.md#d-049) 后 V1 详细交互在本机 Web，飞书"推送 + deeplink 跳 Web"延后到 V1.5 接入。原 [D-022](DECISIONS.md#d-022) 标 partial superseded。

### 故事 2：饭后反馈 —— V2.0

> 14:00，我刚吃完饭回工位。Agent 弹一个迷你卡片：
>
> "中午的清炒胡萝卜肉丝 + 油麦菜 + 米饭怎么样？
>
> 好吃度: ⭐⭐⭐⭐⭐
> 整体满意: ⭐⭐⭐⭐⭐
> [偏油] [分量小] [配送慢] [想再来]
> 备注：（可选）"
>
> 我点 4 星，勾选"想再来"。下次这家就会优先被推。

**关键体验点**：5 秒内完成，不破坏工作节奏。

### 故事 3：临时调整 —— V2.1

> 18:00，今天加班晚下班，想吃顿好的。
>
> 我对 Agent 说："今天想吃辣的、解馋点。"
>
> Agent 重新推 3 个，全是中辣以上的下饭菜。

**关键体验点**：自然语言追加约束，不需要点选项。

### 故事 4：长期反馈进入推荐 —— V2.2

> 用了两周后某天，Agent 推荐的 3 个里有 2 个都是带汤水的炖煮（潮汕牛肉、酸菜鱼）。
>
> 我去 `learned_profile.yaml` 看，确实**统计聚合**显示我对"潮汕煮"和"川菜炖"采纳率最高、平均评分最高。这是统计画像在起作用——不是 LLM 编出来的"洞察"，是数得出来的偏好。决策见 [D-026](DECISIONS.md#d-026)。

**关键体验点**：用得越久越懂你，且**懂的逻辑可解释、可手编**。

### 故事 5：同事接入 —— Phase 2

> 同事 B 在北京中关村办公区。看我用得不错，问要怎么用。
>
> 我告诉他：
> 1. `pip install chisha-data-beijing-zgc`（按工区拆包，他不需要装深圳数据）
> 2. 把 `chisha` 加进自己的 OpenClaw（推荐层）
> 3. 写自己的 profile.yaml
>
> 半小时后他也在用了，但他用的是 GPT-4 不是 Claude，数据存在他自己的服务器，反馈也只他自己看得到。

**关键体验点**：本地闭环，数据隐私彻底；按工区拆包，按需订阅。

## 6. 北极星指标

**唯一最重要的指标：连续使用情况。**

V1 跑通后看两件事：

| 指标 | V1 目标（自用 1 周） | V2 目标（自用 1 月） |
|---|---|---|
| 7 日连续工作日采纳率（推 3 中选 1 的比例） | ≥ 50% | ≥ 60% |
| 餐后好吃度 ≥ 4 星比例（V2 才有反馈） | — | ≥ 70% |
| 周内主动开启 Agent 推荐次数 | ≥ 8 次/周 | ≥ 10 次/周（中晚 × 工作日 5） |

**为什么不再用"决策时间从 15min → 1min"**：这个指标 V1 阶段不可度量、易作弊（用户自己改流程不算产品功劳）。决策见 [D-028](DECISIONS.md#d-028)。

**不优先的指标**：

- 用户体重变化、卡路里数字（受太多因素影响，不可控）
- 营养素精确度（方法论是"结构正确"不是"绝对精确"）
- LLM API 成本（V1 单次推荐 < ¥0.1，月度 < ¥10，可忽略）
- 推荐延迟（5s 还是 30s 不影响"主动推"的体验）

## 7. 不做的事（明确边界）

避免功能蔓延：

| 不做 | 为什么不做 |
|---|---|
| 卡路里精确追踪 | 太重，违背"轻量决策"定位 |
| 严格 1/2-1/4-1/4 餐盘硬约束 | 中式外卖现实下不可达，改弱约束（[D-023](DECISIONS.md#d-023)） |
| 在家做饭推荐 | 数据形态完全不同，先聚焦外卖场景 |
| 食材采购建议 | 同上 |
| 体重 / 体脂记录 | 已有大量专业 APP，不重复造轮子 |
| 训练计划生成 | 是另一个产品，可以未来独立做 |
| 商家平台对接（直接下单） | 各 APP 接口封闭，靠跳转就行 |
| 社交分享（"晒午餐"） | 偏离工具属性 |
| ~~Web/小程序 APP 形态~~ | ~~定位是 Agent Skill~~ → **[D-049](DECISIONS.md#d-049) 翻案**：V1 主交互改本机 localhost Web SPA，飞书降级为 V1.5 推送通道 |
| 独立公网 APP / SaaS 多租户 | 见 [D-001](DECISIONS.md#d-001)（D-049 的 localhost 单用户不属于此项） |
| 数据采集（爬取/清洗/打标） | 由 sister project `chisha-collector` 负责，本仓只消费数据（[D-027](DECISIONS.md#d-027)） |
| LLM 蒸馏 learned_insights | N=1 数据下容易过拟合假规律，改统计聚合（[D-026](DECISIONS.md#d-026)） |

## 8. 成功是什么样

### Phase 1 成功（V1，~6 周）

- 自己每天用 ≥ 1 顿
- 7 日工作日连续采纳率 ≥ 50%
- 飞书卡片在 OpenClaw 里能稳定推送
- 我愿意拍着胸脯把它推荐给同事

### Phase 2 成功（V2，~3 个月）

- 至少 3 个同事在自己 Agent 里跑起来（覆盖至少 2 个工区）
- 反馈系统进入闭环，统计聚合的偏好画像让推荐越来越准
- 出现至少一个"我没想到 Agent 居然推得这么准"的瞬间

### Phase 3 成功（开源后，~6-12 个月）

- GitHub star ≥ 100
- 至少 3 个外部 PR（功能贡献或数据贡献）
- 至少 1 个其他城市的数据 fork
- 出现一个我没接入过的 Agent 形态用户（比如某人接到了 Telegram bot）

## 9. 数据来源边界（必读）

本项目**不负责数据采集**。数据采集由 sister project `chisha-collector` 维护：

- 采集源、清洗、打标、保鲜（暂定 15 天周期）由 collector 负责
- 本仓 `chisha` 只消费 collector 输出的 `restaurants.json` + `dishes_tagged.json`
- 数据格式由两边约定 schema（见 [DESIGN.md §5.2](../DESIGN.md)）
- Collector 与 chisha 各自独立演进、独立发版

详见决策 [D-027](DECISIONS.md#d-027)。
