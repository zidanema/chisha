EVAL DESIGN · 评测集设计说明

菜品打标评测集是怎么构建的？

这份文档讲清楚一件事：我们为菜品打标做的这份 171 条评测集，评测的是什么、为什么这么设计、它能告诉我们什么、不能告诉我们什么。

受众　产品经理 / 业务方 / 评测使用者 篇幅　约 15 分钟阅读 配套　README.md / CRITICAL_RULES.md
TL;DR — 一句话讲清楚

我们做的不是「考模型对菜品理解准不准」的评测集，而是「当我们的规则和大模型的常识不一样时，模型能不能按我们的规则来打标」的评测集。

报告里 deepseek-flash 字段准确率 88.9% 这个数字，不代表生产环境里 88.9% 的菜会被打对，而是「在我们刻意挑出来的难题上，它能拿 88.9%」。

这个数字的用途是 选模型（哪个更听话），不是 预估生产质量（生产真实准确率多少）。

01这件事的业务背景

1.1 菜品打标在做什么

我们的推荐链路需要对每一道菜（如「蒜蓉空心菜」「蜜汁叉烧饭」）打 15 个结构化字段：

  • 菜系 / 主料类型 / 烹饪方式
  • 蛋白克数 / 蔬菜比例 / 含油等级
  • 是否完整一餐 / 辣度 / 角色（主菜/主食/配菜/...）
  • 是否含加工肉 / 甜酱等级 / 含水量 / 主食类型 / 标签

这些字段直接喂给下游推荐和健康评估算法。字段打错 = 推荐错 = 用户看到不合适的菜。

1.2 为什么需要一份评测集

我们要在多个候选模型里选一个跑生产，这些模型的价格差 10-30 倍：

模型100 万条菜品打标成本角色
Claude Sonnet 4.6~$2,000baseline / 上限参考
Claude Haiku 4.5~$600Claude 性价比档
DeepSeek V4 Pro~$120DeepSeek 旗舰
DeepSeek V4 Flash~$100DeepSeek 性价比天花板
Kimi K2.6~$500中文优势候选
为什么不能拍脑袋

如果便宜的模型够用，一年能省几十万。但「够用」这件事不能靠主观判断——需要一份标准答案，对每个模型跑同一批菜，看谁打得最准。这份标准答案就是评测集（Golden Set）。

02核心概念：什么是「Anchor」（锚点）

理解评测集设计，必须先理解 anchor 是什么。它不是"重要案例"的意思，而是"反直觉的规则"。

2.1 一个具体的例子

看这道菜：

「蜜汁叉烧饭」
字段 大模型的"本能"答案 我们规则要求的答案
processed_meat_flag
是否加工肉		 true —— 叉烧听起来像加工肉 false —— 中式烧腊是整块鲜肉熟制，营养上不算加工肉
sweet_sauce_level
甜酱等级 0-3		 0 —— 没看到「糖」字 3 —— 看到"蜜汁"必须打 3
grain_type
主食类型		 可能选「精制面」 白米 —— 米饭就是白米

红色那栏（模型本能）和绿色那栏（我们规则）冲突的地方，就是 anchor。

2.2 为什么会有这种冲突

我们做的是 营养健康评估 业务，不是大众美食推荐。从营养学角度看：

  • 「叉烧不是加工肉」—— 因为它是整块鲜肉烤的，不像培根那种重组乳化工艺
  • 「红烧菜默认含糖」—— 红烧工艺普遍加冰糖/老抽（含糖色），15-30g 糖/份很常见
  • 「关东煮汤底不喝」—— 钠含量超标，含水量算 2 不算 3

这些专业判断和大众语义不一样。大模型训练数据里 99% 是大众语义，所以它有强烈的"本能错误倾向"。

2.3 四个最容易冲突的字段

我们项目里发现这 4 个字段是 anchor 密度最高的：

字段模型本能规则要求
sweet_sauce_level 把"非显式甜"打 0 看到「烧/红/酱/糖/蜜/照」字眼就打 ≥ 2
processed_meat_flag 把"听起来工业化的肉"打 true 只有"腊/培根/午餐肉/丸料"才打 true
dish_role 按字面理解 7 项优先级 + 触发词清单（套餐 > 饮品 > 汤 > 主食 > 主菜 > 配菜 > 小食）
grain_type 按形状归类 "米粉/河粉/肠粉/红薯粉"都归白米（精制淀粉）
关键结论

anchor = 模型本能 vs 规则 冲突的判断点。评测集的核心目标，就是测模型能不能压住自己的本能，按规则来打。

03设计思路：评测集是怎么挑菜的

3.1 反例：如果用「真实订单抽样」会怎样

最直觉的做法是：从真实外卖订单里 随机抽 171 条菜，让人工标好正确答案，跑评测。听起来很合理，但实际跑出来的结果会是：

真实抽样评测集（假想）

— 按真实出现频率抽 171 条 —
模型字段准确率
Sonnet 4.698.2%
Haiku 4.597.8%
DeepSeek Flash97.5%
Kimi K2.697.0%
差异不到 1pp，决策不出来——到底选谁？

Anchor 加权评测集（实际）

— 按"能踩多少 anchor"加权抽 171 条 —
模型字段准确率
Sonnet 4.689.4%
DeepSeek Flash88.9%
Haiku 4.585.3%
Kimi K2.682.1%
差异 7pp，决策成立：选 DeepSeek Flash

为什么真实抽样会"挤在一起"？因为真实订单里大部分菜是「白米饭、可乐、番茄炒蛋、葱花鸡蛋汤、青菜豆腐」这种简单 case，所有模型都能 99%+ 答对。简单 case 把模型差异稀释掉了。

3.2 我们的做法：Anchor 加权选菜

核心原则：选菜的时候，按"这条菜能踩到多少个 anchor"来选，不按"真实出现频率"来选。

菜真实外卖出现频率选/不选为什么
白米饭极高不选0 个 anchor，区分不了模型
可乐极高不选同上
番茄炒蛋高不选字段太平凡
蜜汁叉烧饭中选一条菜踩 3 个 anchor
麻辣香锅 单/双人份中选测 processed_meat 边界 + 价格推理
一次性筷子套装低选测非食物兜底（spicy=0 即使是调料）
恰巴塔三明治 4 件套极低选测幻觉防御（历史上被瞎编成"烤鲜活鲍鱼"）

3.3 一个类比：驾照路考

类比

驾照路考不会考你「在空旷停车场直行 10 米」—— 所有人都会。

它会考「侧方位停车」「坡道起步」「连续变道」—— 这些是司机能力的分水岭。

通过路考 ≠ 你能在所有路况下开好车，但代表你在有挑战的场景下也能应对。

→ 我们的评测集就是这种"路考卷子"。它不测简单场景，专测有难度的场景。

05这份评测集能回答什么 / 不能回答什么

这是最容易踩坑的地方。请仔细看下面的对比。

能回答

Q1：6 个候选模型里，哪个最适合跑生产？

已得出结论：DeepSeek Flash（距冠军 0.5pp，便宜 20 倍）。

Q2：模型在 4 大易错字段上的能力差距有多大？

报告里有「关键字段分水岭」单独切片。

Q3：哪些菜系/anchor 模型最容易错？

报告里有「按类别切片」。

Q4：换 prompt 版本后，新 prompt 是变好还是变差？

能回答（前提：评测集本身的 anchor 集没变）。

不能回答

Q1：生产环境里，deepseek-flash 真实打标准确率是多少？

不能直接回答。88.9% 是难题加权下的，生产真实应该更高，可能 95%+，但具体多少未知。需要另建"真实抽样校准集"。

Q2：用户在前端会感知到多少打标错误？

不能回答。要看下游推荐链路对各字段错误的敏感度，是另一个评测话题。

Q3：模型对 prompt 里没写到的边界 case 推理能力如何？

部分回答。当前评测集主要测「规则服从」，不专门测「规则泛化/类推」。

5.1 用三个类比帮你理解定位

这份评测集像它不是
驾照路考一年实际驾驶记录
公务员考试工作绩效考核

它评估「能力上限」和「关键弱点」，不评估「日常表现的平均值」。

06横评报告里的数字怎么读

报告（report.md）里有这些数字，业务方读的时候请按这个理解。

6.1 字段级准确率（如 deepseek-flash 88.9%）

正确理解

「在我们这套难题加权评测集上，该模型有 88.9% 的字段答对」。

错误理解（要避免）
  • × 「该模型生产环境会有 88.9% 准确率」—— 不对，生产真实会更高
  • × 「11.1% 的菜会打错」—— 不对，是 11.1% 的"字段-菜"组合在难题集上打错

6.2 整条全对率

171 条菜里，15 个字段全部打对的比例。这个数字会比字段级准确率低得多（如果字段级 88.9%，整条全对可能只有 30-50%），因为只要任一字段错就算全条错。

→ 整条全对率更接近"严格质量门"的概念，但也不是生产指标，仍然是难题加权下的数字。

6.3 4 大字段单独切片（最重要！）

sweet_sauce_level / processed_meat_flag / dish_role / grain_type 单独看准确率。

模型选型的关键决策点

这是模型选型最关键的一组数字。如果某模型整体 85%，但 4 大字段都 < 70%，意味着它在最重要的判断点上很弱，不能用。

6.4 JSON 合法率

模型有没有按要求输出合法的 15 字段 JSON。这是及格门槛——< 95% 直接淘汰，因为下游解析会大面积失败。

07这份评测集怎么演进（待办）

7.1 短期：补一份真实抽样校准集 [P1]

目的：回答「生产真实准确率到底是多少」。

方法：

  • 从生产真实菜品里随机抽 100-200 条
  • 按真实频率分布（不加权 anchor）
  • 人工标注 expected
  • 跑 deepseek-flash 得到「生产真实准确率」

这个数字会和当前 88.9% 形成配对：

  • 当前 88.9% 用于模型选型（拉开差距）
  • 校准集 95%+（假设）用于对外汇报生产质量

7.2 中期：用户反馈反哺 [P2]

V1 推荐链路上线后，从 user feedback 里挑「模型打标错了的菜」补进评测集——从 anchor-driven 转向 feedback-driven。

KNOWN_ISSUES.md 里已经留了 16 条边界争议条目，等 user feedback 触发重审。

7.3 长期：能力分项 [P3]

把当前混在一起的能力拆开评测：

  • 规则服从能力（当前评测集主测）
  • 规则泛化/类推能力（prompt 没写到的近邻 case）
  • 幻觉防御能力（菜名怪/含 emoji/超长促销词）
  • 价格推理能力（同名不同价的 protein 估算）

每项一个子评测集，目的是定位"这个模型在哪类能力上弱"，便于换模型或改 prompt 时精准调整。

给业务方的一句话总结

这份评测集是「卷子」，不是「画像」。

它告诉你"DeepSeek Flash 是 6 个候选里最适合的"——这是它的核心价值，已经为公司省了几十万的模型成本。

它不告诉你"生产环境里打标质量好不好"——这是另一个问题，需要另一张卷子。

看到 88.9% 这个数字，请理解为"它在我们设计的难题上的能力上限"，不要理解为"用户感知到的质量"。
